# 02. 设计目标分解 (Objective Decomposition)

> **方法论级别**: Senior Designer Standard
> **核心命题**: 把模糊的 PRD 翻译成"可衡量、有优先级、能约束后续所有决策"的三层目标体系

---

## 定义与目的

设计目标分解，是把"产品想要什么"拆解成三类彼此独立又相互制约的目标——**业务目标、用户目标、体验目标**——并明确它们之间的冲突、权衡与优先级的推理过程。

它不是把 PRD 的"项目背景"抄一遍，而是回答一个尖锐的问题：**这次设计成功的标准到底是什么，谁来验收，用什么数字验收。**

**资深设计师与初级设计师的本质区别**：

- **初级设计师**：看到 PRD 写"做一个让用户方便管理订单的后台"，就把"方便管理订单"当成目标，直接开始画原型。
- **资深设计师**：会追问三层——
  - 业务为什么要做这个后台？（降低客服人力成本 30%）
  - 用户用它要达成什么？（5 分钟内处理完一批退款，不用切 3 个系统）
  - 体验上什么算好？（新客服无培训 10 分钟上手，误操作率 < 1%）

再举一个电商例子。PRD 写"优化购物车，提升转化"。

- **初级**：目标 = "优化购物车体验"。这句话无法验收——什么叫优化？多好算好？
- **资深**：拆成
  - 业务目标 BG：购物车到下单转化率从 38% 提到 45%（Q3 前）
  - 用户目标 UG：用户能一眼看清"凑满减还差多少"，不用心算
  - 体验目标 EG：加购到下单点击步数从 5 步降到 3 步

**目标分解的目的，是产出一份能"反向约束"后续所有设计决策的判据**：

1. 当 IA、流程、交互出现多个方案时，用目标当裁判（哪个更服务北极星指标，选哪个）
2. 当需求蔓延时，用目标当边界（不服务任何目标的功能，砍掉）
3. 当团队吵架时，用目标当共识（争论"好不好看"变成争论"是否达成 EG"）

**这是整个设计推理的"指南针"**——任务建模、IA、流程、交互全都要回头对齐这里。目标错了或缺了优先级，后面每一步都在精确地走向错误方向。

---

## 适用场景

- ✅ **从 0 到 1 的产品/大版本**：方向未定，必须先锚定目标（新业务线、重构）
- ✅ **多方利益冲突的项目**：业务、技术、用户诉求打架，需要显式权衡（中后台、平台型产品）
- ✅ **目标模糊的 PRD**：PRD 只写了功能没写成功标准，需要反推（最常见场景）
- ⚠️ **纯需求迭代**：单点功能优化，目标继承自上层，只需局部对齐而非全量分解
- ⚠️ **创意/品牌项目**：业务目标清晰但体验目标偏感性，需结合品牌策略方法
- ❌ **纯 Bug 修复 / 文案微调**：无目标决策空间，直接做即可，分解是过度设计

## 输入要求

| 输入 | 必需 | 说明 |
|------|------|------|
| requirement_inventory | ✅ | 需求清单 / PRD，目标的原料来源 |
| business_context | ✅ | 业务背景（所处阶段、商业模式、当前核心矛盾），决定目标取向 |
| user_persona | ⭕ | 用户画像（如有），帮助提炼真实用户目标 |
| metrics_baseline | ⭕ | 现有数据基线（如有），决定目标能否量化、目标值是否现实 |
| stakeholder_input | ⭕ | 关键干系人诉求（如有），用于识别隐性目标和冲突 |

**输入质量决定输出质量**：

- 如果 PRD 只有功能没有成功标准，必须把推断出的目标标 `inferred: true`，并在 `traceable_to_prd` 写 `[inferred]`
- 如果没有 metrics_baseline，目标值会变成"拍脑袋"，需在 rationale 说明取值依据（行业基准/同类产品/历史经验）
- 如果业务背景缺失，无法判断"快速上线"还是"完美体验"优先，必须先问，不要发明

- `business_goals` / `user_goals` / `experience_goals` / `design_constraints` / `scope_boundaries` 是 schema required，缺一不可
- 每个 business_goal 必须有可量化的 `success_metric`（带数字、带时间），否则不算目标
- 每个推断目标必须 `inferred: true` 且 `traceable_to_prd` 标 `[inferred]`
- experience_goals 的 dimension 限定在 Nielsen 可用性维度枚举内（efficiency / learnability / memorability / error_prevention / satisfaction / accessibility / performance）

## 推理过程

### Step 1: 区分三层目标（核心专业判断）

**这是资深与初级的第一道分水岭**：很多人把三类目标混成一锅粥。

| 维度 | 业务目标 (Business) | 用户目标 (User) | 体验目标 (Experience) |
|------|--------------------|-----------------|----------------------|
| 回答 | 公司/业务要什么 | 用户要完成什么 | 怎样算"做得好" |
| 主语 | 业务/组织 | 用户角色 | 产品界面 |
| 度量 | GMV、成本、留存、转化 | 任务完成、痛点消除 | 步数、时长、错误率、满意度 |
| 例子 | 退款成本降 30% | 5 分钟处理完一批退款 | 误操作率 < 1% |

**资深设计师的思考**：

三层是**手段-目的链**，不是平行清单：
- 体验目标好 → 用户目标达成 → 业务目标实现。
- 反向检验：每个体验目标必须能回答"它服务哪个用户目标/业务目标"；服务不了的，就是设计师的自嗨。

**专业方法**：对每条 PRD 需求做"三问归类"——
1. 这是"业务想要的结果"吗？→ 归业务目标
2. 这是"用户想完成的事"吗？→ 归用户目标（用 JTBD 句式：当……时，我想……，以便……）
3. 这是"界面层面的好坏判据"吗？→ 归体验目标（必须可测量）

**示例（CRM）**：
```
PRD: "让销售更高效地跟进客户"
  → 业务目标 BG: 团队成交率 +20%（Q4）
  → 用户目标 UG: 销售一眼看到今天该跟进谁，不漏单
  → 体验目标 EG: 进入系统到看到待跟进列表 ≤ 2 秒、0 点击
```

**Junior 错误**：
- ❌ 把"做一个客户管理模块"当成目标（这是功能，不是目标）
- ❌ 三层混写：把"提升转化"和"按钮要明显"放在同一层
- ❌ 体验目标写成感性词（"界面要简洁好看"）而非可测量判据

---

### Step 2: 从模糊 PRD 提取可衡量目标（量化能力）

**PRD 90% 不会写成功指标。资深设计师的职责是把形容词翻译成数字。**

**专业方法——"形容词 → 指标"四步**：

1. **抓住形容词/模糊词**："高效""方便""提升""优化""流畅"
2. **追问可观测行为**："高效"体现为什么行为？→ 处理时长、点击步数
3. **找基线**：现状是多少？（有数据用数据，无数据用同类产品/行业基准，并标注来源）
4. **定目标值 + 时间**：从 X 到 Y，截止 T

**示例（电商）**：
```
PRD 原文："优化结算流程，让下单更顺畅"
  形容词："顺畅"
  可观测行为：结算页流失率、填写耗时、报错次数
  基线：结算页流失率 42%（取自现有埋点）
  量化目标：BG-003 结算页流失率 42% → 30%，Q3 前
            EG efficiency: 填写时长从 90s → 45s
            EG error_prevention: 表单报错率 < 5%
```

**关于"无法量化"的目标**：
- 真正无法量化的目标（如"提升品牌信任感"），不要硬编个假数字。改为"代理指标 + 验证方法"——如"信任感"代理为"NPS"或"是否愿意绑定银行卡的比例"，并在 measurement_method 写清。
- 宁可写"代理指标 + 主观验证（5 人访谈）"，也不要写"用户感觉很好"这种不可证伪的目标。

**Junior 错误**：
- ❌ 把 PRD 形容词原样搬进目标（"让操作更方便"）
- ❌ 编造无依据的目标值（"转化提升 50%"——凭什么？）
- ❌ 遇到难量化的就放弃量化，写一句空话

---

### Step 3: 识别目标冲突（资深设计师的洞察）

**目标之间天然打架。junior 假装它们和谐，senior 主动把冲突挖出来摆桌面。**

**最常见的四类冲突**：

1. **快速上线 vs 完美体验**：工期 2 周，但好引导需要 4 周打磨
2. **业务增长 vs 用户体验**：业务要加弹窗促活，用户要清净（如电商首页塞满促销 vs 浏览效率）
3. **覆盖全面 vs 聚焦核心**：想服务所有角色 vs 资源只够做透一个角色
4. **转化 vs 留存**：诱导下单提转化，但伤害长期信任降留存

**专业方法**：
- 两两扫描业务目标与体验目标，问"达成 A 是否会损害 B"
- 冲突必须写成结构化 `goal_conflicts`：goal_a / goal_b / 冲突描述 / **权衡建议** / 决策人 / 状态
- 关键：**权衡建议要给方向，不能甩锅**。"这个让产品决定"不是权衡，"建议首版保 A、B 降级到 v1.1，因为……"才是。

**示例（CRM）**：
```
CONF-001
  goal_a: BG 两周上线抢客户（快速上线）
  goal_b: EG 销售零培训上手（完美体验）
  冲突: 工期不够做新手引导
  权衡建议: 首版牺牲引导，保上线；用"3 个示例数据 + 帮助气泡"做轻量过渡，
            完整引导排 v1.1。理由：抢占窗口期 > 首版体验完美度。
  决策人: 业务负责人
```

**Junior 错误**：
- ❌ 假设所有目标都能同时达成（goal_conflicts 为空）
- ❌ 识别了冲突但不给权衡建议，原样抛给产品
- ❌ 权衡只凭个人偏好，不说理由（无法被评审）

---

### Step 4: 优先级排序与北极星指标

**目标平均用力 = 没有重点 = 必然顾此失彼。**

**专业方法——两步定优先级**：

1. **先定北极星指标（North Star）**：在所有业务目标里，选**唯一一个**最能代表"产品为用户创造的核心价值"的指标。
   - 它不是越多越好的虚荣指标（如总注册数），而是价值兑现指标（如周活跃创建数、完成核心任务的用户数）
   - 例：CRM 北极星 = "每周有效跟进的客户数"（既反映用户在用，又预示成交）
   - 例：电商北极星 = "完成首购的新用户数"（而非 GMV，GMV 可被大客户扭曲）

2. **按"对北极星的贡献度"给目标排 P0/P1/P2**：
   - P0：直接驱动北极星，不做产品就不成立（通常 1-2 个）
   - P1：支撑 P0 或解决关键痛点
   - P2：锦上添花，资源够才做

**资深判断细节**：
- P0 数量是质量信号。**全是 P0 = 没有 P0**。如果分解出 6 个 P0，说明还没真正取舍。
- 优先级要能向上追溯：每个 P0 都能说清"它如何推动北极星"。

**Junior 错误**：
- ❌ 不设北极星，所有业务目标并列
- ❌ 把所有目标标 P0（领导都说重要）
- ❌ 北极星选了虚荣指标（注册数、PV）而非价值指标

---

### Step 5: 用 scope_boundaries 防止目标过多

**目标过多不是在 Step 4 才解决，而是要显式画出"不做什么"。**

**专业方法**：
- `in_scope`：本期为达成 P0 目标必须做的（从 P0 目标反推，而非从功能列表勾选）
- `out_of_scope`：**明确写出不做的**——这比 in_scope 更重要。junior 只写做什么，senior 敢写"这些不做"
- `future_consideration`：好想法但本期不做，挂起来防止它反复冒出来打断聚焦
- `scope_risks`：识别哪些目标容易引发蔓延（如"顺便把报表也做了吧"），预设挡回话术

**关键认知**：每多一个目标，资源就从 P0 上分走一块。**砍目标的能力，和提目标的能力同样资深。** 一个只会往里加目标、从不敢往外砍的设计师，是 junior。

**Junior 错误**：
- ❌ out_of_scope 为空（什么都想做）
- ❌ 不识别范围蔓延风险
- ❌ 把"未来可能做"和"本期要做"混在一起

---

### Step 6: 用目标约束后续所有决策（资深设计师的真正价值）

**目标分解的产物不是一份文档，而是一把贯穿全流程的尺子。**

**资深设计师如何用目标做裁判**：

| 后续环节 | 用目标怎么裁 |
|---------|-------------|
| 任务建模 | 任务必须可追溯到某个 user_goal；追溯不到的任务砍掉 |
| 信息架构 | 北极星相关信息放最高层级，P2 目标的功能下沉 |
| 页面流程 | EG efficiency「核心任务 ≤ 3 步」直接约束流程不能超 3 步 |
| 交互设计 | EG error_prevention「误操作率 < 1%」决定要不要二次确认 |
| 方案评审 | 两个方案选谁？选更服务北极星 + 不违反约束的那个 |

**专业方法**：在每个目标里埋好"约束钩子"——
- 体验目标的 `target` 要写成可直接当验收标准的句子（"≤ 3 步""< 2 秒"），让下游能拿来即用
- 业务目标的 `related_user_goals` 建立追溯链，下游任何产物都能回溯到目标

**Junior 错误**：
- ❌ 目标定完就锁进抽屉，画原型时全凭手感
- ❌ 体验目标写得无法当验收标准（"体验流畅"——怎么验收？）
- ❌ 后续方案与目标无关联，评审时只能比"谁画得好看"

## 质量标准

### Must Have（资深设计师必做）

1. ✅ 三层目标分离且完整：业务 / 用户 / 体验各至少 1 条，归类正确
2. ✅ 每个业务目标有可量化 `success_metric`（带数字 + 时间），不可证伪的目标不算目标
3. ✅ 每个用户目标用 JTBD 句式描述真实诉求，关联到角色
4. ✅ 每个体验目标 dimension 合法、target 可测量、有 rationale 说明服务哪个上层目标
5. ✅ 设定唯一北极星指标，且为价值指标非虚荣指标
6. ✅ 优先级分层真实（P0 不超过 2-3 个），每个 P0 能追溯到北极星
7. ✅ 至少识别 1 组目标冲突，并给出带理由的权衡建议
8. ✅ scope_boundaries 同时有 in_scope 和 out_of_scope，out_of_scope 非空
9. ✅ 所有推断目标标 `inferred: true` 且 traceable_to_prd 标 `[inferred]`

### Should Have（资深设计师应做）

10. ✅ 业务目标与用户目标建立 related 追溯链
11. ✅ 体验目标覆盖多个 Nielsen 维度（不只 efficiency）
12. ✅ 难量化目标提供代理指标 + 验证方法，而非放弃量化
13. ✅ 识别范围蔓延风险（scope_risks）并预设应对

### Nice to Have（加分项）

14. ⭐ 给出 success_criteria 的三维（业务/用户/技术）整体验收标准
15. ⭐ 目标值附行业基准/同类产品对标依据
16. ⭐ 冲突标注 decision_maker 和 resolution_status，形成决策闭环

## 失败模式

### FM-1: 目标=功能复述

**现象**：目标写的是"做一个 XX 模块""支持 XX 功能"

**识别信号**：
- business_goals 里出现功能名词而非结果指标
- success_metric 为空或写"上线该功能"
- 整份目标无一个数字

**根因**：停留在"要做什么"，没追问"做成了算什么"

**修复**：对每条目标问"它达成时，哪个数字会变化"——答不出就不是目标

---

### FM-2: 目标不可衡量

**现象**：目标全是"提升""优化""更方便"等形容词

**识别信号**：
- success_metric 无数字、无时间
- 体验目标 target 是感性描述（"简洁美观"）
- 无法回答"怎么验收"

**根因**：缺少把形容词翻译成指标的量化能力

**修复**：走 Step 2 四步法，找基线、定目标值与时限；难量化的用代理指标

---

### FM-3: 假装目标不冲突

**现象**：goal_conflicts 为空，所有目标"都能达成"

**识别信号**：
- 既要快速上线又要完美体验，却无冲突记录
- 业务诉求与体验诉求并列无张力
- 项目中后期才暴露"做不完"

**根因**：回避矛盾，或没有能力预判冲突

**修复**：两两扫描 BG×EG，主动找"达成 A 是否伤害 B"，写进 goal_conflicts

---

### FM-4: 优先级平均主义

**现象**：所有目标都是 P0，或没有北极星

**识别信号**：
- priority 全 P0
- 无 North Star，业务目标并列
- 资源无法聚焦，每个都做一半

**根因**：不敢取舍，或把"领导都说重要"当优先级依据

**修复**：先定唯一北极星，按对北极星贡献度排序，强制 P0 ≤ 2-3 个

---

### FM-5: 目标过多无重点

**现象**：分解出十几个目标，out_of_scope 为空

**识别信号**：
- scope_boundaries.out_of_scope 缺失或为空
- 目标清单越列越长
- future_consideration 和 in_scope 混在一起

**根因**：只会加不会砍，缺少范围管理意识

**修复**：从 P0 反推 in_scope，显式写 out_of_scope，把"好但不急"挪进 future_consideration

---

### FM-6: 目标与下游脱节

**现象**：目标定完锁进抽屉，后续设计全凭手感

**识别信号**：
- 体验目标无法当验收标准
- 任务/IA/流程与目标无追溯关系
- 评审时只能比"好不好看"

**根因**：把目标当交付文档，而非贯穿流程的尺子

**修复**：体验目标写成可直接验收的句子，建立 related 追溯链，评审时用目标当裁判

---

## Anti-Patterns

1. ❌ **功能伪装目标**：把"做 XX 模块"当目标，而非"做成后哪个数字变化"
2. ❌ **形容词目标**：满篇"提升/优化/流畅"，无一个可验收的数字
3. ❌ **目标和谐幻觉**：假装所有目标都能同时达成，从不记录冲突
4. ❌ **全 P0 主义**：所有目标都最重要，等于没有重点
5. ❌ **虚荣北极星**：用注册数/PV 当北极星，而非价值兑现指标
6. ❌ **只加不砍**：out_of_scope 永远为空，目标越堆越多
7. ❌ **甩锅式权衡**：识别了冲突却只说"让产品定"，不给带理由的建议
8. ❌ **抽屉目标**：目标定完就不再用，后续决策与目标零关联

## 与其他方法的关系

### 上游依赖

- **01-input-diagnosis**：输入诊断的需求清单与业务背景，是目标分解的原料；诊断出的缺口决定哪些目标需要标 `inferred`

### 下游消费

- **03-user-task-modeling**：design_objectives 是任务建模的目标锚点，任务必须可追溯到 user_goal
- **04-scenario-modeling**：用户目标 + 体验目标界定场景的成功条件
- **07-information-architecture**：北极星与优先级决定 IA 的信息层级
- **08-page-flow-modeling**：体验目标的 efficiency target 直接约束流程步数
- **17-quality-rubrics / 19-traceability**：目标是质量评分与全链路追溯的根节点

**关键传导**：

- 北极星指标 → 决定 IA 谁在最高层级
- 体验目标的可测量 target → 直接成为流程/交互的验收标准
- 目标优先级 → 决定资源往哪倾斜、范围在哪收口

**目标是指南针**——它不产出界面，但决定后面每一步往哪走。缺了它，团队会"高效地走向错误方向"。

### 北极星指标

**North Star = 客服日均独立处理完成的退款工单数**。它同时反映效率（BG-001）、上手度（BG-003）和真实价值兑现（用户真的被服务到），而非"功能上线数"这种虚荣指标。

### 这个输出为什么是"资深级别"

1. ✅ **三层分离且服务链清晰**：每个 EG 都标了服务哪个 BG/UG，没有自嗨目标
2. ✅ **把形容词翻译成了数字**："更高效"→8min→3min，"上手快"→2天→0.5天
3. ✅ **主动暴露并权衡了冲突**：CONF-001 快速上线 vs 完美引导，CONF-002 效率 vs 安全，都给了带理由的方向
4. ✅ **优先级真实**：只有 2 个 P0，且都追溯到北极星
5. ✅ **敢写 out_of_scope**：报表、绩效看板、规则引擎明确不做，并预设了挡回蔓延的话术
6. ✅ **推断目标全部标注**：BG-001/BG-003/UG-002 标 inferred，不冒充 PRD 原文
