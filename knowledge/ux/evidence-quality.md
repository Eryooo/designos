# 证据质量标准（Evidence Quality）

> 通用资产 · `ux.evidence-quality` · status: pilot
> 让体验问题从主观描述变成可沟通证据的通用标准。本文规定证据的质量属性,
> 不规定具体采集工具、文件格式或存储路径。

## purpose

确保每条体验结论都有可追溯、可复现、可定位的物证。没有证据的问题在评审会上无法推动,也无法区分事实、推断与主观偏好。

## applies_to

- 体验问题清单、可用性测试报告、启发式评估报告。
- 任何需要把 issue 绑定到页面/流程/元素的场景。

## decision_framework

证据类型选择:
- `screenshot`:单页面视觉/状态问题,默认优先。
- `dom`:结构性问题,如 Tab 顺序、对比度、语义。
- `trace`:流程性问题,如多页面跳转、长操作。
- `video`:动画/时序问题,仅在静态截图无法表达时使用。

证据充分性三属性:
1. **可追溯**:指向具体页面/流程/元素。
2. **可复现**:按描述路径可再次到达同一现象。
3. **可定位**:能被结论稳定引用,最好有稳定 evidence id 与局部定位。

交付资格:
- `final_delivery_ready`:证据足够支撑最终问题清单。
- `fallback_safe`:只够输出受限中间结果。
- `supplement_required`:必须先补关键证据。
- `blocked`:完全无法继续。

## senior_heuristics

- 关键交互组件至少覆盖能证明问题的状态对比(如 default + hover/focus/error)。
- 分辨率低、无说明、无 OCR/文本线索、命名混乱时,不要交付"看起来完整"的问题清单。
- 高风险结论优先要流程证据(trace)或多步截图,不要只靠单张截图。
- 证据弱的结论要标推断,不与事实结论混写。

## quality_rubric

| 分级 | 信号 |
|---|---|
| 优 | 每条 issue 有 evidence_refs,证据可追溯/复现/定位,关键状态覆盖充分 |
| 中 | 多数 issue 有证据,但部分状态缺失或定位模糊 |
| 差 | 大量 issue 无证据,截图命名混乱且无法关联结论 |

## common_failure_modes

- **证据断链**:issue 写得很明确,但 evidence_refs 为空或指不到具体页面。
- **截图噪音**:截图含无关插件/广告/敏感信息,评审无法聚焦。
- **状态缺失**:只截默认态,却判断 hover/focus/error 问题。
- **过度自动化信任**:工具采到图不等于证据足够,仍要判断是否支撑结论。

## do_not_claim

- 不规定截图/DOM/trace 的存储路径。
- 不定义某个工具的采集实现。
- 不保证证据本身真实无误,只规定证据质量判定口径。
