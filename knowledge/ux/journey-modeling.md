# 旅程建模（Journey Modeling）

> 通用资产 · `ux.journey-modeling` · status: pilot
> 把"功能页面视角"升级为"用户体验旅程视角"的通用方法。本文与任一产品无关,
> 只给结构、判定规则与反模式。

## purpose

解决一个核心痛点:评估若按页面/菜单顺序做,会忽略页面之间的衔接体验。旅程建模以用户视角串联多页面,描绘完成一个目标的全过程,为后续任务生成与问题归因提供阶段锚点。

## applies_to

- 需要从端到端任务视角发现衔接问题的评估。
- 需要把问题归因到"用户在哪个阶段受阻"的场景。

## decision_framework

通用六阶段模板(不是每个产品都全有,建议至少覆盖前四):
1. 进入与定位 → 2. 前置配置 → 3. 任务执行 → 4. 结果理解 → 5. 报告决策 → 6. 闭环优化。

建模步骤:
1. **选主角**:挑 1–2 个核心角色(频率最高 + 业务影响最大),不给所有角色建旅程。
2. **列阶段**:每阶段写"用户试图做什么"+ 参与角色;跨角色阶段写多角色。
3. **标关键路径**:每阶段标 1–3 个关键任务(高频 + 业务影响大 + 易出问题)。
4. **标情绪曲线**(可选,有访谈数据时):为严重等级判定提供基线。
5. **标体验机会点**:每阶段 1–2 个"最易出问题的位置"。

## senior_heuristics

- 至少有一条跨角色旅程(哪怕"提交 → 审批"两步),协作衔接最易藏问题。
- 旅程命名用"动词 + 名词"(端到端规则上线),不用"XX 模块使用"。
- 每个体验机会点后续都应产生对应评估任务,否则机会点是空话。
- 有情绪基线再标严重等级,等级判定不容易飘。

## quality_rubric

| 分级 | 信号 |
|---|---|
| 优 | 阶段反映真实用户目标,含跨角色衔接,机会点都落到后续任务 |
| 中 | 阶段合理但全是单角色,或机会点未被后续消费 |
| 差 | 按菜单顺序建阶段(IA 不是旅程),或超过 8 阶段粒度过细 |

## common_failure_modes

- **把 IA 当旅程**:"工作台 → 数据管理 → 设置"是信息架构,不是旅程。
- **角色割裂**:给每个角色建独立完整旅程,丢掉协作交织。
- **阶段过细**:超过 8 阶段,失去主线。
- **无情绪基线就定级**:严重等级判定漂移。

## do_not_claim

- 不规定旅程的输出 schema(消费方自定)。
- 不绑定任何 skill 的 stage 编号。
- 不替代真实用户旅程访谈,情绪曲线在无数据时为推断。
