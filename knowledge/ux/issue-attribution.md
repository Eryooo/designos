# 问题归因（Issue Attribution）

> 通用资产 · `ux.issue-attribution` · status: pilot
> 把 raw issue 转成结构化、可评审、可修复的问题条目的通用方法。

## purpose

将问题从"现象描述"升级为"归因清楚、证据可追、影响明确、建议可执行"的结构化 issue。归因维度包括旅程阶段、任务、模块、原则、严重等级与证据。

## applies_to

- 启发式评估、可用性测试、设计验收中的问题清单生成。
- 需要把问题合并、去重、排序、进入修复队列的场景。

## decision_framework

归因步骤:
1. **归因到旅程阶段**:按截图页面、原则适用模块、问题文本中的操作匹配 journey stage;无法归因时标未归因并报警比例。
2. **归因到任务**:能标尽量标,用任务上下文解释影响。
3. **归因到模块**:必填;跨模块问题归到全局模块。
4. **判严重等级**:按 `ux.severity-rubric`。
5. **写用户影响**:结构为"在某场景下,某角色做某事时,遇到某现象,导致某影响"。
6. **写建议**:改什么 / 改成什么 / 为什么。
7. **标来源基础**:需求/实现证据/通用启发式推断。
8. **去重合并**:同原则同模块合并;跨模块同类合并到全局;已知问题标 known。

## senior_heuristics

- 每条 issue 至少有一个主原则与一条证据;否则很容易混入功能 bug 或主观意见。
- suggestion 不应淹没主报告;高价值问题优先。
- 用户影响缺角色/场景/动作/后果任一项,都不足以推动修复。
- required_actions 控制在少数关键项,否则团队无法执行。

## quality_rubric

| 分级 | 信号 |
|---|---|
| 优 | issue 可直接进入修复评审:证据、影响、严重等级、建议都闭环 |
| 中 | 信息大体齐全,但部分建议不够可执行或归因偏粗 |
| 差 | raw issue 罗列,缺证据/缺影响/缺建议/缺原则 |

## common_failure_modes

- **功能测试偏移**:把功能存在与否当作体验问题。
- **无原则 issue**:没有 principle_id 的问题进入主清单。
- **无证据 issue**:evidence_refs 为空仍交付。
- **过细碎**:同类问题不合并,主报告超过 50 条。
- **影响空泛**:写"影响用户体验"但没有场景和后果。

## do_not_claim

- 不定义具体 Issue schema 字段名。
- 不保证自动归因 100% 正确,归因失败比例高时需要人工校准。
- 不替代修复方案设计稿。
