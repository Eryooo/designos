# uxeval Client Mode 跨工具可用度评估

日期：2026-05-25  
对比对象：
- Trae IDE + `designos v0.4.0`
- Claude Code + `designos v0.4.0`

输入条件：
- 同一份 PRD
- 同一组 26 张截图
- 同为 client mode

## 一句话结论

当前问题已经基本可以判定：`这不是某一个承载工具的问题，而是 uxeval client mode 在这类输入条件下，还达不到你要求的 90%+ 可用度。`

更直接地说：
- Trae 跑出来覆盖率 `16.7%`
- Claude 跑出来关键页面覆盖率 `25%`，P0 覆盖率 `0%`
- 两个工具都没有达到“替代低阶任务”的标准

所以当前瓶颈不在：
- Trae
- Claude Code
- 某个单独模型的提示词执行习惯

而在：
- client mode 的证据获取能力
- 输入契约设计
- 核心主链路截图覆盖机制

## 这说明了什么

### 1. 现在的 client mode 已经会“安全失败”，但还不会“高质量完成”

这件事本身有价值，因为它至少不再瞎交付。

但从你的目标看，这还远远不够。你的目标不是：
- 会跑
- 会拦住风险
- 会提示补资料

你的目标是：
- 尽量自动完成
- 尽量少打断用户
- 最终结果能替代一部分低价值设计工作

在这个标准下，当前 client mode 只能算：
- `可信`
- 但`不够强`

### 2. 这次对比已经足够证明：问题不主要在工具，而在模式

如果是某个工具的问题，通常会出现这种情况：
- 一个工具明显更高覆盖
- 一个工具明显更会自动补证据
- 一个工具明显更接近 final delivery

但这次没有。

相反，这次两边都指向同一个结论：
- 缺的就是核心增量学习主链路截图
- P0 页面没有被真正覆盖
- 系统更多是在做“受限评估”，不是完整评估

这说明当前最需要改的不是“换个 IDE 再试”，而是：
`重构 client mode 如何拿到足够的关键证据。`

## 跨工具结果对比

| 维度 | Trae v0.4.0 | Claude Code v0.4.0 | 结论 |
|------|-------------|--------------------|------|
| 最终状态 | `supplement_required` | `fallback_safe` | 两者都未达到最终交付 |
| 置信度 | Low | 受限可交付 | 两者都不够强 |
| 关键页面覆盖率 | 16.7% | 25% | 都明显过低 |
| P0级覆盖率 | 未达到完整覆盖 | 0% | 核心问题一致 |
| 核心功能覆盖率 | 明显不足 | 20% | 都不满足目标 |
| 人工介入次数 | 0 | 0 | 自动化有了，但没完成任务 |
| 结果质量 | 结构化补料和阻断较强 | fallback 报告更完整 | 都更像受限交付，不是最终交付 |

## 解决了什么问题

当前 V1 已经解决了这些问题：

- 不会在证据不足时伪装成 final report
- 不会强迫用户先批量重命名截图
- 能自动提取 PRD、识别覆盖率、发现冲突、生成补料单
- 能自动输出 fallback / supplement / clarification 类结构化产物

这些都很重要，但它们解决的是：
`可信度和边界控制问题`

还没有解决的是：
`自动化完成度问题`

## 不解决会出现什么情况

如果沿用当前 client mode 继续直接对外测试，你会持续遇到下面这些情况：

- 工具看起来很聪明，但经常停在 fallback / supplement
- 设计师仍然要补关键主链路截图
- 核心评估价值仍然依赖人工准备输入
- 最终自动化只能替代“整理和初筛”，不能替代“完整低阶评估执行”

也就是说，当前它能替代的主要还是：
- PRD 结构化理解
- 截图 inventory
- 初步问题归类
- 补料规划

而不是：
- 对核心功能主链路做高覆盖的完整体验评估

## 当前最核心的系统性问题

### P1. client mode 没有拿到“关键页面优先”的足够强输入

现在的问题不是截图数量少，而是：
- 关键截图不对
- 关键状态不全
- 核心主链路覆盖过低

这说明当前 client mode 即使做了：
- input planning
- auto-remediation
- draft mapping

它仍然没有把“拿到关键页面和关键状态”这件事做成足够强的前置能力。

### P1. 自动化流程已经闭环，但自动化产出质量还没闭环

当前系统已经能自动执行：
- PRD 抽取
- 截图分析
- 冲突识别
- 问题输出
- 交付审计

但这条闭环更像：
`会自动判断自己为什么现在还不够`

而不是：
`会自动把自己推到足够好的结果`

### P1. 当前 client mode 更适合“资料驱动受限评估”，不适合“主力完整评估”

这次对比已经很说明问题：

- 之前的 browser / web 自动化结果，更擅长发现真实运行态问题
- 现在的 client 模式，更擅长在资料不充分时安全地收口

所以当前更合理的产品定位不是：
`client mode = 主力完整评估模式`

而更像：
`client mode = 离线资料评估 / 补证据 / 受限评估模式`

## 这对产品规划意味着什么

### 长期正确方案

以下方向是长期正确的：

- 保留 current gate / audit / fallback 机制
- 保留低摩擦输入策略，不把命名当门槛
- 保留 auto-remediation / proactive planning / draft mapping

这些不该回退。

### 当前不够的地方

如果你希望 client mode 也达到至少 `90%+` 可用度，那下一阶段不能再只修“风险控制”，而必须补：

1. 更强的关键页面获取机制
2. 更强的关键状态识别机制
3. 更强的 screenshot-description 自动对齐
4. 更强的“首轮输入就尽量够用”的约束
5. 更强的混合模式能力，例如：
   - client 输入不足时，允许引导切到 web mode
   - 或者生成最小补采集清单，而不是泛化补料

## 结论：当前版本到底到了什么程度

如果按你要的标准判断：

- `当前 client mode 还不能算 90%+ 可用`
- `也不能算已经足够替代低阶任务`
- `但已经证明了它不会再瞎交付，并且有清晰的工作流底座`

所以当前最准确的定性是：

`V1 的 client mode 目前已具备可信的离线受限评估能力，但还不具备高覆盖、高完成度的主力自动评估能力。`

## 下一步建议

不要再把问题归因到 Trae 或 Claude Code。  
这轮对比已经足够说明：`真正要迭代的是 client mode 产品能力，而不是继续换承载工具。`

下一步最值得做的，不是继续横向测更多工具，而是先决定：

### 路线 A：接受 client mode 的定位

把它正式定义为：
- 离线资料评估
- 补证据工作流
- fallback / bounded 交付模式

而把真正高质量完整评估交给 web mode。

### 路线 B：继续把 client mode 往主力模式做强

那就必须补一轮更高杠杆能力：
- 关键主链路自动补证据
- 更强的 OCR/页面识别
- 更强的关键状态抽取
- 更强的补采集指引

一句话总结：

`这次 Trae + Claude Code 的对比已经证明，当前瓶颈不是工具，而是 client mode 还没有强到足以成为高可用主力评估模式。`
