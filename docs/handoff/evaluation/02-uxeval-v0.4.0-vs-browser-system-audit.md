# uxeval v0.4.0 对比浏览器自动化结果系统审计

日期：2026-05-25  
对比对象：
- Trae IDE 客户端运行的 `uxeval v0.4.0`：`/Users/young/Documents/Codex/designosv0.4.0`
- Codex 浏览器自动化历史结果：`/Users/young/Documents/Codex/data-design-review`

## 一句话结论

`uxeval v0.4.0` 这次在 Trae 里已经证明了“边界控制”和“补料/降级机制”是成立的，但还没有证明它在这个真实案例里达到了“可替代低价值设计工作”的最终质量标准。

更直接地说：
- 它现在已经不会乱交付假完整报告了，这是明显进步。
- 但它这次拿到的核心证据太少，最终只达到了 `supplement_required`，没有达到真正可交付的 `final_delivery_ready`。
- 和之前 Codex 浏览器自动化结果相比，这次更像“安全但保守的受限评估”，不是“高覆盖、高置信度、可直接替代”的完整评估。

## 这次修复已经解决了什么问题

### 1. 解决了“看起来完成、实际不可信”的风险

之前最危险的问题不是能力弱，而是系统可能在证据不足时继续产出形式完整的报告。  
这次 `v0.4.0` 没有这么做，而是明确停在：
- `supplement_required`
- `Fallback Package`
- `Low confidence`

这说明当前技能已经具备生产系统最重要的一个基础能力：`证据不足时不冒充最终结论。`

### 2. 解决了“截图命名和人工整理负担过重”的问题

这次结果里没有把“必须先重命名大量截图”当成准入门槛，而是把重点放在：
- 缺哪些关键页面
- 缺哪些关键状态
- 缺哪些核心模块截图

这符合产品方向。否则 DesignOS 就会把低价值整理劳动重新甩回设计师。

### 3. 解决了“只能阻断，不能给有价值中间产物”的问题

当前 `v0.4.0` 不只是停下来，它还能给出：
- fallback report
- supplement request
- 结构化的 evidence / issue / audit 中间产物

这说明它已经从“会报错的工具”升级成了“有工作流语义的评估系统”。

## 当前还存在的核心问题

### P1. 这次结果还不能算“可替代低价值设计工作”的最终交付

最关键的问题不是代码，而是这次真实运行结果本身。

在 `/Users/young/Documents/Codex/designosv0.4.0/uxeval_run/outputs/README.md` 里，当前结论明确是：
- `Delivery Status: supplement_required`
- `Confidence: Low`
- `截图覆盖率: 16.7%`

这意味着：
- 它没有完成最终交付
- 它也没有达到你定义的正常模式近 `99%-100%` 标准
- 它甚至连高质量 fallback 都还没有完全证明

这不是系统乱来，而是系统正确地承认：`当前输入不足以支撑最终高质量结论。`

从产品角度看，这是一种“正确失败”，不是“成功交付”。

### P1. 这次输入对核心业务链路覆盖太弱，系统主要证明了“会拦住风险”，没有证明“能自动完成任务”

从同一个 README 和 `uxeval_report.md` 看，这次缺失的关键页面非常集中：
- 增量学习节点配置界面
- 制作 Pickle 模型弹窗
- 制作增量学习模型弹窗
- 版本列表弹窗
- 算法参数配置界面（方案2）

同时截图分布也明显偏离主目标：
- 控制台/管理中心 14 张，占 58%，低相关
- 任务流建模只有 1 张
- 模型库只有 2 张，且还是空状态

这说明当前 `v0.4.0` 在这个案例里，更多是在做：
- 输入规划
- 证据缺口识别
- 风险阻断

而不是完成：
- 核心增量学习主链路的高质量体验评估

这和“替代低价值工作”的终极目标还有距离，因为真正的目标应该是：
`系统先自动拿到尽可能完整的关键证据，再自动产出高质量评估，而不是主要靠用户补核心截图。`

### P1. 相比之前浏览器自动化结果，当前 client 模式在真实运行态问题发现上明显偏弱

在 `/Users/young/Documents/Codex/data-design-review/outputs/数据挖掘V3.9.4_启发式评估问题汇总_全量版.md` 里，之前浏览器自动化结果做到的是：
- 31 个问题
- 19 个高优先级
- 17 个真实环境确认问题
- 3 个空间覆盖
- 2 个主路径异常
- 5 类系统性结论

它不仅能指出“缺什么”，还能指出真实运行态里的问题，例如：
- 空间错跳
- 错误落入训练空间
- 模型训练页空白

而这次 `v0.4.0 client` 结果更多还是：
- 验证现有截图
- 识别缺哪些关键证据
- 把部分问题标成 verified / unverified

这说明：
- `client 模式` 当前已经更安全了
- 但在“真实运行路径诊断能力”上，仍明显弱于之前的浏览器自动化模式

换句话说：  
当前 `client` 更像“安全的证据审计员”，还不像“高覆盖的自动化体验评估员”。

### P2. 结构化产物已经有了，但给下游自动评判和复用的契约还不够稳

Markdown 输出已经比较清楚，但这次部分 stage JSON 的契约仍然不够稳定，表现为：
- 有些关键字段不在预期顶层
- issue / audit 结构存在较强的嵌套依赖
- 一些 machine-readable 字段不够易于直接消费

这对人看报告影响不大，但对你后面想做的两件事影响很大：
- 让另一个 skill creator 大模型直接判断输出质量
- 让更多 skills 共享同一套 evidence / issue / delivery contract

所以从 OS 平台角度看，这 still 是个要补的点。

### P2. “已验证问题”和“基于 PRD 推断的问题”之间的边界仍需更严

当前 `uxeval_report.md` 里有一些问题虽然被放进主结论，但从证据链看仍然更接近：
- 局部截图证据 + PRD 推断
- 或证据不够完整下的较强推测

这一步已经比以前好很多，因为系统至少把很多问题列进了 `unverified_issues`。  
但如果目标是正常模式接近 `99%-100%`，那“verified”门槛还应该再硬一点。

## 和之前浏览器自动化结果对比后的总体判断

### 当前 v0.4.0 的优势

- 更会控制边界，不会乱交付
- 更会识别输入缺口
- 更会给结构化补料单
- 更符合“命名是加速项，不是门槛”的产品方向
- 更适合做可暂停、可恢复、可审计的 client 工作流

### 当前 v0.4.0 的不足

- 对这次真实案例，核心主链路证据仍不足
- 自动补救还没有强到足以把这次 run 拉到 final_delivery_ready
- 对真实运行态问题的发现能力仍弱于 web / browser 自动化
- 结构化 JSON 契约仍未完全收敛成稳定平台协议

## 现在这版到底到了什么程度

如果从“DesignOS 能否替代一部分低价值设计工作”来判断，我会这样定级：

- 已达到：
  - `L2-L3 之间`
  - 可以做内部试点
  - 可以明显减少截图整理、补料判断、证据归档、初步体验评估结构化输出的人工成本

- 还没达到：
  - 不能对这个案例宣称“生产级 final 评估已可自动完成”
  - 不能对 client 模式宣称“已普遍达到近乎满分的自动交付质量”

更准确的说法是：

`当前 v0.4.0 已经证明了 client 模式的工作流底座是对的，但在这个真实案例里，它还没有证明自己已经足够强到可以独立完成高质量最终评估。`

## 为什么这不是失败

这次结果不是“没做成”，而是暴露了一个非常关键的产品真相：

`client 模式的上限，强依赖输入证据质量和系统自动补证据能力。`

这次它至少做对了两件最值钱的事：
- 没把低质量输入伪装成高质量输出
- 很清楚地告诉你，离高质量最终交付还缺什么

这其实说明当前 V1 已经摆脱了“危险的假智能”，进入了“保守但可信”的阶段。

## 下一步最值得做的事

不是继续无边界修小点，而是围绕两个方向做下一轮产品判断：

### 方向 1：验证 web 模式是不是你真正的高质量主力模式

因为从这次对比看：
- `web/browser 自动化` 更擅长发现真实运行问题
- `client` 更擅长做受控、可审计、低摩擦的证据工作流

这可能意味着未来正确的产品定位不是“二者二选一”，而是：
- `web` 做高质量主评估
- `client` 做离线/资料驱动/补证据/受限评估

### 方向 2：强化 client 模式的“自动补证据能力”

如果你仍希望 client 模式尽量逼近替代标准，那么真正该补的不是更多 gate，而是：
- 更强 OCR
- 更强 screenshot-description 自动对齐
- 更强关键页面/关键状态识别
- 更强自动补料规划
- 更强 draft mapping 的高置信度提升能力

一句话总结：

`当前 v0.4.0 已经证明它不会再瞎交付，但还没有证明它在这个真实案例里已经足够强到独立完成最终高质量评估。`
