# Client Mode 90%+ 可用度升级路线

日期：2026-05-25  
目标：把 `uxeval client mode` 从“可信的受限评估模式”升级成“90%+ 可用度的主力客户端评估模式”

## 一句话结论

如果要把 `client mode` 做到你要的水平，核心不是继续加更多 gate，也不是继续优化报告文案，而是要把产品模型从：

`分析用户已经给出的截图`

升级成：

`主动规划、主动引导、主动补强关键证据，再完成高质量评估`

也就是说，Route B 的本质不是“让 AI 更会看图”，而是：

`让系统在无法自动操作客户端的前提下，尽量把关键证据获取这件事产品化、最小化、结构化。`

## 当前真正卡住用户和产品的是什么

### 已解决的问题

当前 V1 已经解决了这些基础问题：
- 不会瞎交付假完整报告
- 会做 evidence gate / delivery audit
- 会做 auto-remediation / proactive planning / draft mapping
- 不会把截图命名当准入门槛
- 能给 fallback / supplement / clarification 结果

这些都很重要，但它们主要解决的是：

`结果可信度问题`

### 还没解决的问题

当前真正挡住 90%+ 可用度的，不是报告生成，而是这 3 件事：

1. `关键证据获取能力不够强`
2. `关键页面/关键状态识别能力不够强`
3. `系统还没有把“最小补采集任务”产品化到足够低摩擦`

所以当前 client mode 会出现这样的局面：
- 自动化流程很完整
- 风险控制也很好
- 但核心主链路覆盖还是低
- 最终更像“会判断自己不够”，而不是“会把自己推到够用”

## Route B 的正确目标

### 不是这个

- 不是让 client mode 完全不需要用户提供任何截图
- 不是让系统在证据不足时也硬出结论
- 不是让用户做大量重命名、手工 mapping、手工解释

### 而是这个

在客户端应用无法被 Agent 自动操作的前提下：

1. 系统先自动理解 PRD 和目标任务
2. 系统自动生成最小关键采集计划
3. 用户只补最少量、最高价值的截图/状态
4. 系统自动完成大部分证据对齐、覆盖判断、问题识别、交付审计
5. 最终达到 90%+ 可用度

所以这里最关键的一句话是：

`client mode 的主任务不是“评估任意截图”，而是“用尽可能小的人工补采集，换取足够高的核心评估覆盖”。`

## Route B 的产品核心改造

### 核心改造 1：从“截图输入”升级为“关键证据采集任务”

这是整个 Route B 最重要的一刀。

当前系统还是偏：
- 用户给截图
- 系统判断够不够

这不够。

应该升级成：
- 系统先根据 PRD 自动生成 `Capture Mission`
- 明确告诉用户：这轮评估只缺哪 5-8 张关键截图、哪 3-4 个关键状态
- 用户补的是“最小关键任务”，不是“泛化补资料”

这会极大减少用户的无效劳动。

### 核心改造 2：把“关键页面 / 关键状态”识别做成主能力

当前的问题不是截图数量不够，而是：
- 关键页面没拿到
- 关键状态没拿到
- 核心业务链路没有闭环

所以 Route B 的重点不该是继续泛泛提高 OCR，而是：
- 识别哪些页面是 P0 / P1
- 识别哪些状态是必须覆盖
- 确定哪些证据足以支撑最终交付

### 核心改造 3：把“最小人工介入”做成产品约束

Route B 的用户介入应该只剩两类：

1. `系统客观拿不到的关键截图/关键状态`
2. `系统无法自动判断的高层业务决策`

不应该再有这些低价值工作：
- 批量重命名
- 手写完整 screens-map
- 手写长篇截图说明
- 多轮来回补充“你再给我一些图看看”

## 升级路线

## 阶段 1：Capture Mission V2

### 目标
把当前的 input planning 从“缺什么”升级成“最小关键采集任务单”。

### 要做的事
- 输出 `must_capture_pages`
- 输出 `must_capture_states`
- 输出 `nice_to_have_pages`
- 输出每张建议截图的采集目的
- 输出最小通过线：达到什么覆盖即可进入高质量评估

### 成功标准
- 用户第一次看到的不是泛泛补料，而是一张很短的关键采集任务单
- 同类 run 中，中途 pause 次数显著减少

## 阶段 2：Evidence Fusion V2

### 目标
把 OCR、文件名、markdown、目录结构、draft mapping 融成更强的单一证据判断层。

### 要做的事
- 强化 screenshot-description 自动对齐
- 强化 page/state 多证据打分
- 明确 high / medium / low 的业务含义
- 只有高置信度映射才能帮助 final delivery

### 成功标准
- 大部分常规截图集无需手写 mapping
- 歧义图只剩少量 clarification

## 阶段 3：Client Critical Path Coverage Engine

### 目标
把“核心功能主链路覆盖率”做成第一目标，而不是事后统计指标。

### 要做的事
- 从 PRD 自动提取关键主链路
- 为每条主链路定义必须页面和必须状态
- 运行时优先围绕主链路而不是泛化截图做 coverage 计算
- P0 覆盖不足时，不允许把通用表单问题冒充完整评估结果

### 成功标准
- 最终报告对“主链路是否覆盖”有确定性结论
- P0 缺失会被更早发现、更早收口

## 阶段 4：Bounded Clarification UX

### 目标
把人工介入压缩成一次、少量、结构化确认。

### 要做的事
- clarification package 只列最少量歧义项
- 用户只需补几张关键图或确认几条映射
- 同一 run 对同一缺口不重复打断

### 成功标准
- 平均人工打断次数趋近 0 或 1
- 用户补的是最少量关键信息，而不是整理工作

## 阶段 5：Client Final Delivery Hardening

### 目标
把“90%+ 可用度”真正落实到交付标准。

### 要做的事
- 正常模式只允许高置信度、高覆盖度、高证据完备度进入 final report
- fallback 模式只允许输出 85%+ 可支持内容
- 主问题清单和待验证问题严格分层

### 成功标准
- final_delivery_ready 的误放行接近 0
- fallback 不再看起来像“差一点的最终报告”，而是边界清晰的受限结果

## 这条路线的 V1.5 结束线

Route B 不能再无限修下去，所以必须有结束线。

我建议的结束线是：

1. 在一组真实 client benchmark 上：
   - 关键页面覆盖率平均达到 `90%+`
   - P0 页面覆盖率平均达到 `90%+`
   - 正常模式最终交付中，不出现明显误放行

2. 人工介入负担：
   - 平均人工打断次数 `<= 1`
   - 不要求大规模重命名
   - 不要求手写完整 mapping

3. 产物可用度：
   - final report 在高质量输入场景下可直接用于评估交付
   - fallback 在低质量输入场景下也边界清晰、可信

## 不该继续做的方向

如果你走 Route B，下面这些方向要避免：

- 继续堆更多 gate，而不增强证据获取
- 继续把更多责任甩给用户补资料
- 继续强化截图命名要求
- 继续让用户写完整 screens-map
- 继续只优化报告包装，不解决核心证据问题

## 我的明确建议

下一轮不要再泛化成“继续提升 client mode”，而是按下面这个顺序推进：

1. `Capture Mission V2`
2. `Evidence Fusion V2`
3. `Critical Path Coverage Engine`
4. `Bounded Clarification UX`
5. `Final Delivery Hardening`

一句话总结：

`Route B 要成功，client mode 就必须从“截图评估器”进化成“关键证据采集与评估系统”。`
