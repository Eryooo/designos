# 数据挖掘体验启发式评估 Skill 封装交接总览

## 1. 先回答你的判断

你说的“执行一步生成一步的执行脚本”，**不完全对，但也不算错**。

更准确的说法是：这次执行里实际存在 **4 层产物**，其中只有第 3 层会表现得像“边执行边修”。

1. **评估任务定义层**  
   先基于 PRD + 截图产出“要评什么”。
   - 文件：`数据挖掘V3.9.4_启发式评估任务清单_体验导向版.md`

2. **真实环境执行脚本层**  
   再把任务定义翻成“从哪个页面开始、点哪里、看什么、何时截图、何时记问题”。
   - 文件：`数据挖掘V3.9.4_启发式评估执行脚本_真实环境版.md`

3. **运行时桥接与校准层**  
   真正进入真实环境后，会遇到新页签、iframe、错跳空间、遮罩层、空白页、只读边界等问题。这里不是重新发明评估逻辑，而是在**校准执行器**。
   - 文件：`scripts/playwright_bridge.mjs`
   - 特征：这一步最像“执行一步，修一步”

4. **问题沉淀与汇总层**  
   每个空间单独沉淀问题底稿，最后归并成业务可发的总表。
   - 文件：`空间B_...问题清单.md`
   - 文件：`空间C_...问题清单.md`
   - 文件：`数据挖掘V3.9.4_启发式评估问题汇总_全量版.md`

所以，**不应该把整个过程理解成“AI 每点一步都重新生成脚本”**。  
真正应该封装成 skill 的，是：

- 固定的评估流水线
- 固定的运行边界
- 固定的证据和问题记录结构
- 固定的自动化桥接能力
- 固定的异常处理策略

而不是把当前某一次具体点击轨迹硬编码成 skill。

## 3. 我对 skill 封装的建议结论

### 结论一：不要把它做成“单一大 skill”

这次过程同时包含：

- PRD 解析
- 静态评估
- 真实环境自动化
- 手动登录接管
- 多空间证据归并
- 业务报告输出

如果全塞进一个巨型 `SKILL.md`，很快会失控，触发条件也会变脏。

更合理的做法是拆成两层：

1. **核心 skill：`web-heuristic-eval-readonly`**
   - 通用能力
   - 可复用到别的 Web 产品
   - 重点解决“只读自动化评估”和“问题沉淀”

2. **项目资产包：`data-mining-v394-eval-pack`**
   - 当前项目的 PRD、截图、空间策略、输出模板、命名规范
   - 不做成 skill 触发器，只做 repo 内资源

### 结论二：最该固化的是“边界”和“状态机”

这次执行最贵的不是点击动作本身，而是这些边界判断：

- 什么时候先做静态评估，不该直接上 Playwright
- 什么时候生成执行脚本，而不是直接开跑
- 什么时候必须等用户手动登录
- 什么时候只允许填表不允许提交
- 什么时候是产品问题，什么时候是环境异常
- 什么时候应该记“待验证”，不能硬判体验问题

这些才是 skill 的真正价值。

### 结论三：把“桥接脚本”当工具，不要当 skill 本体

`scripts/playwright_bridge.mjs` 应该被视为：

- skill 的底层执行器
- 低自由度、可扩展的 deterministic tool

而不是把它的实现细节直接堆进 `SKILL.md` 正文。

## 4. Claude Code 实现时的目标

Claude Code 接手后，不应该只做到“能跑一次 demo”，而应该做到：

1. 用户提供 PRD 和截图时，可以稳定产出：
   - 任务清单
   - 执行脚本
   - 问题底稿
   - 汇总报告

2. 用户提供测试环境地址并手动登录后，可以稳定跑：
   - 只读模式
   - 多空间模式
   - 新页签 / iframe / 错跳页面
   - 可填但不提交

3. 自动化过程失败时，不把失败伪装成“没有问题”，而是：
   - 明确标记为环境异常
   - 输出阻塞点
   - 更新问题清单或执行记录
