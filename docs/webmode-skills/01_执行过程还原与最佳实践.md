# 执行过程还原与最佳实践

## 1. 真实执行过程不是一条线，而是 6 个阶段

### 阶段 A：项目收敛

先做的不是自动化，而是收敛项目边界：

- 只做体验启发式评估
- 不做设计还原度验收
- 输入以 `PRD + 截图 + 真实环境` 为准
- 输出以问题清单和汇总报告为准

**最佳实践**：
- 先把范围说死，再动自动化
- 先删旧业务残留，再谈复用方法论

**如果不这样做的后果**：
- 自动化会继承旧业务的错误前提
- 评估目标会在“设计验收”和“体验评估”之间来回摇摆

---

### 阶段 B：静态理解与任务生成

先读 PRD、截图，输出两份关键文档：

1. **任务清单**：定义“应该评什么”
2. **PRD-截图冲突清单**：定义“哪些点不能直接下结论”

这一步不需要真实环境。

**最佳实践**：
- PRD 是主基准，截图是现实校准层
- PRD 和截图冲突时，显式记录冲突，不要擅自抹平
- 先拆“用户任务”和“观察点”，再写执行脚本

**不要做的事**：
- 不要刚看到截图就直接写问题清单
- 不要把截图里暂时看不到的能力判成缺失

---

### 阶段 C：执行脚本化

在静态任务清单之上，再生成一份真正给自动化/人工执行用的脚本。

这份脚本必须明确：

- 起始页面
- 每一步点哪里
- 目标页面是什么
- 到达后看什么
- 哪些场景必须截图
- 发现问题后记哪些字段

**最佳实践**：
- “任务清单”和“执行脚本”分开
- 执行脚本必须写到页面/控件/状态级
- 多空间策略要在开跑前定死

**不要做的事**：
- 不要把抽象的“体验观察点”直接当运行脚本
- 不要把多空间选择留到执行中间再临时决定

---

### 阶段 D：运行时桥接和环境接管

真实环境执行时，核心不是 prompt，而是桥接能力。

这次真实打通的能力有：

- 新页签切换
- iframe 切换
- DOM 定位和点击
- 填表但不提交
- 节点拖拽但不保存
- 截图与文本回读

**最佳实践**：
- 用户手动登录，自动化接管已登录会话
- 只读模式必须是执行器内建边界，不是口头约束
- Playwright 层支持 `eval`，因为复杂系统里单纯文本点击不够稳

**不要做的事**：
- 不要把账号密码写入聊天或脚本
- 不要默认每个页面都能通过 `getByText` 稳定点击
- 不要假设卡片点击一定是同页跳转

---

### 阶段 E：空间级问题沉淀

这次没有把所有问题都先写进一个总表，而是按空间拆底稿：

- 主观察空间：主链路问题
- 空状态空间：低数据/无数据引导问题
- 入口异常空间：空间错跳、空白页等主路径异常

**最佳实践**：
- 每个空间单独一份问题底稿
- 同时记录“执行到哪一步”和“为什么停”
- 环境异常也记成问题或执行阻塞，不要吞掉

**不要做的事**：
- 不要把“自动化没点进去”直接当“页面没问题”
- 不要把不同空间的证据混成一个表，最后再回忆来源

---

### 阶段 F：业务汇总

最后再把各空间底稿归并成业务可发版总表。

这份总表不是流水账，而是：

- 执行摘要
- 当前结论
- 业务优先级建议
- 归并问题清单
- 重点问题详情
- 按来源拆分的完整问题记录

**最佳实践**：
- 汇总文档要先做“系统性归并”，再附完整台账
- 业务方优先看“为什么要先修这个”，不是先看所有底稿编号

---

## 2. 这次真正沉淀下来的最佳实践

### 2.1 先静态，后动态

真实环境执行之前，必须先有：

- 任务清单
- 执行脚本
- 冲突清单

否则自动化只会变成漫游。

### 2.2 把“执行边界”固化为规则

这次边界是：

- 可以看、点、填、拖
- 不可以保存、提交、执行、发布、注册、导入导出

这是 skill 必须固化的安全规则。

### 2.3 环境异常本身也是结果

例如：

- 空间入口错跳
- 进入后页面空白
- 覆盖层拦截点击

这些都不该被自动化吞掉，而应该沉淀成：

- 问题记录
- 执行阻塞说明
- 待验证结论

### 2.4 不要把问题记录和业务汇总混在一起

建议始终保留两层：

1. **底稿**
   - 细
   - 按空间
   - 按执行时序

2. **汇总**
   - 面向业务方
   - 做归并和优先级
   - 有建议方向

### 2.5 Playwright 不是唯一难点，真正难点是页面拓扑

这次最影响效率的不是 click API，而是：

- 控制台卡片开新页签
- 外层壳页套 iframe
- 目标页还能继续开新页面
- 错跳空间导致上下文不可信

所以 skill 里最该被强调的是：

- page/frame state 管理
- 当前上下文校验
- 路由异常识别

## 3. 需要规避的问题

### 3.1 规避“边执行边改 prompt”的假象

这次看起来像不断重写，其实大部分是在补执行器能力。  
封装 skill 时，不要把这种临场调试误解成“prompt 需要无限细化”。

真正应该拆开的是：

- 稳定流程
- 底层工具
- 环境特异性异常

### 3.2 规避把当前业务路径写死进 skill

不能把下面这些直接固化成 skill 逻辑：

- `数据挖掘V394BVT空间`
- `ylgui2自动化51302`
- `数据挖掘V393自测空间1`
- 某个 tenantId
- 某个具体菜单名组合

这些应该是：

- 当前项目输入
- 执行配置
- 任务清单实例

不是 skill 的公共行为。

### 3.3 规避把“产品 bug”误判成“AI 点错”

这次已经真实发生过：

- 目标卡片文本正确
- 自动化 DOM 定位也正确
- 用户手动点击复核后，依然进入了错误空间

所以 skill 必须支持：

- 人工复核记录
- “不是自动化误点”的显式标注

### 3.4 规避无证据硬判

如果页面没进去、截图不够、状态不明，就应该：

- 标成环境异常
- 标成待验证
- 标成阻塞点

而不是直接归为体验问题结论。

### 3.5 规避把导出能力和评估逻辑耦合

报告导出可以单独实现。
不要让“PDF 导不出来”反过来阻塞评估主流程。

---

## 4. 适合固化成 skill 的内容

1. 触发条件
2. 预检流程
3. 输入检查
4. 任务清单生成
5. 执行脚本生成
6. 多空间执行规则
7. 只读模式边界
8. 手动登录接管机制
9. 问题记录格式
10. 汇总报告结构

## 5. 不适合固化进 skill 正文的内容

1. 某次项目的 tenantId / URL
2. 某次项目的空间名
3. 当前业务的具体页面命名
4. 一次性的截图文件路径
5. 某次运行的 session id

这些应该放：

- 执行配置
- 项目输入目录
- 运行时状态文件
