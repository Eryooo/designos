# Claude Code 实现任务清单

## P0：先实现对的架构，不要先润色 prompt

这次 skill 化最容易跑偏的点，是过早沉迷 `SKILL.md` 文案和 prompt 细节。  
正确顺序应该是：

1. 先确定目录结构和职责边界
2. 再确定状态机
3. 再确定脚本能力
4. 最后才细修 `SKILL.md` 和 references

---

## 1. 第一阶段：搭骨架

### 1.1 建 skill 目录

目标：

- 建立 skill 文件夹
- 写最小 `SKILL.md`
- 建 `references/`
- 建 `scripts/`
- 建 `assets/`

验收：

- 目录职责清晰
- 没有把大量实现细节塞进 `SKILL.md`

### 1.2 建状态文件机制

目标：

- 能记录当前 run 状态
- 能记录当前空间进度
- 能记录阻塞原因

验收：

- 支持从中断点恢复
- 不依赖会话记忆继续执行

## 3. 第三阶段：实现主流程

### 3.1 `plan` 流程

输入：

- PRD
- 截图

输出：

- 任务清单
- PRD-截图冲突清单
- 第一轮问题底稿

验收：

- 先有“要评什么”，再有“怎么跑”

### 3.2 `run` 流程

输入：

- 执行脚本
- 环境地址
- 空间配置

输出：

- 空间级问题清单
- 截图证据
- 执行记录

验收：

- 支持手动登录后接管
- 支持只读多空间执行
- 支持环境异常显式记录

### 3.3 `summarize` 流程

输入：

- 各空间问题底稿

输出：

- 全量业务汇总
- PDF

验收：

- 先归并系统性问题
- 再附完整问题台账

## 5. 第五阶段：做 4 个真实验收用例

### 用例 1：只有 PRD + 截图

期望：

- 只跑静态评估
- 不强行要求真实环境

### 用例 2：PRD + 截图 + Web 地址

期望：

- 先出执行脚本
- 再等待用户登录

### 用例 3：多空间只读执行

期望：

- 每个空间单独出底稿
- 最后再合并总表

### 用例 4：路由异常 / 空白页

期望：

- 不假装页面正常
- 显式记录为环境异常或主路径问题

---

## 6. 需要明确告诉 Claude Code 的实现红线

1. 不要把这次项目的空间名、URL、tenantId 写死进公共 skill
2. 不要把“手动登录协作”省掉
3. 不要默认自动化失败就是选择器问题
4. 不要把环境异常吞掉
5. 不要把任务清单和执行脚本混成一个文件
6. 不要只给业务方流水账，不做归并
7. 不要为了做 skill，把 repo 当前已有资产全推倒重来

## 7. 最终交付物建议

Claude Code 最终应该至少交付：

1. 可触发的 skill 目录
2. 一份精简但够用的 `SKILL.md`
3. 一组 `references/`
4. 可执行脚本：
   - bridge
   - state
   - merge
   - pdf export
5. 一组最小可运行示例输入
6. 一份验证记录

如果它只给出一个长 prompt，或者只给出一个漂亮的 `SKILL.md`，但没有：

- 状态机
- 运行脚本
- 异常处理
- 汇总脚本

那就说明它只做了“描述封装”，没有做“可运行封装”。
