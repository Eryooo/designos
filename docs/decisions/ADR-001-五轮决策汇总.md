## R1：运行时架构

| # | 决策 | 结论 | 备选 | 理由 |
|---|---|---|---|---|
| 1.1 | Skill 执行模型 | 模式 C（双模式） | A 纯 LLM / B 纯引擎 | 兼顾 Skills 原生 + Eval/CI 能力 |
| 1.2 | 主语言 | Python 3.11+ | Node.js / Go | LLM 生态中心、Pydantic、DSPy |
| 1.3 | CLI 框架 | Typer | Click / Cobra | 同语言、类型安全 |
| 1.4 | 数据校验 | Pydantic v2 | JSON Schema | 业界标杆、错误友好 |
| 1.5 | Pipeline 配置 | YAML | TOML / Python DSL | 可读、非工程师能改 |
| 1.6 | 测试框架 | pytest | unittest | 标准 |
| 1.7 | 类型检查 | pyright strict | mypy | 速度快、严格 |
| 1.8 | Lint + Format | ruff | black + pylint | 现代、极快 |
| 1.9 | 依赖管理 | uv | poetry / pip | 10× 快、2025 标准 |
| 1.10 | 打包 | hatch | setuptools | 现代化 |
| 1.11 | 日志 | structlog | loguru | 结构化 |
| 1.12 | Trace | OpenTelemetry（M2） | 自建 | 业界标准 |
| 1.13 | MCP 部署 | stdio 默认 + 浏览器 SSE | 全 HTTP | 官方惯例、零运维 |
| 1.14 | 外部 MCP | 声明依赖不重建 | 自建 Figma/MasterGo | 复用生态 |
| 1.15 | 安装模式 | M1 全局安装，M2 加 symlink | 三种同时 | 避免影响 M1 周期 |
| 1.16 | 模型配置 | 用户可配 API Key + BASE_URL | 硬编码 | 跨模型、内网兼容 |

## R2：数据契约 + 信息流

| # | 决策 | 结论 | 理由 |
|---|---|---|---|
| 2.1 | 项目工作区结构 | `runs/001-xxx/` 按序号 | 清晰表达顺序和依赖 |
| 2.2 | Skill 间传递 | 默认提示确认（B），可关闭为显式声明（C） | 平衡自动化与控制感 |
| 2.3 | 产物类型枚举 | Skill 自声明 + Kernel 动态注册 | 可扩展、新 Skill 不改 Kernel |

## 开发规范（M1 铁律）

```
1. pyright strict — 所有函数有类型标注
2. Pydantic Model — 模块间接口，禁止 dict
3. structlog — 禁止 print
4. DesignOSError + 错误码枚举 — 统一异常
5. 环境变量 + YAML — 禁止硬编码
6. 核心路径有单测 + 1 个 E2E 冒烟
7. Conventional Commits + PR review
8. ruff check 全过
```
