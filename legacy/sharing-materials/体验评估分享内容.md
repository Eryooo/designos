**二、01 项目背景与问题**

标题：

**01 项目背景：传统体验评估为什么难以规模化？**

这里要明确讲“原有设计流程的问题”。

传统方式痛点：

- **依赖专家经验**：评估质量高度依赖资深设计师个人经验，新人很难稳定复现。
- **评估口径不统一**：有人偏功能测试，有人偏视觉问题，有人偏交互流程，问题粒度不一致。
- **需求与实现脱节**：评估时常只看页面，不容易系统对照需求目标、用户角色和真实任务路径。
- **问题证据不足**：很多体验问题停留在“感觉不顺”“这里不清楚”，难以推动产品和研发认同。
- **产物难以复用**：评估完成后留下的是问题列表，而不是下一次可复用的方法、模板和流程。
- **设计师价值后置**：设计师往往在页面完成后介入评审，影响力集中在发现问题，而不是前置治理体验质量。

AI-native 解决方案：

- 用 Agent 读取和整理 PRD、原则、截图等输入。
- 用用户旅程重构评估路径，而不是按功能菜单机械检查。
- 用启发式原则约束评估口径。
- 用截图证据、问题描述、严重等级和解决方案形成闭环。
- 把高级设计师经验沉淀成团队可执行的标准方法。

这一部分的关键句：

**体验评估 Agent 的价值，不是替设计师判断好坏，而是把“如何判断”变成可执行流程，把“为什么是问题”变成可沟通证据。**

**三、02 UXEval Agent 协同工作流**

标题：

**02 Agent 工作流：从输入资料到体验问题闭环**

建议做成 6 步流程，和参考材料的 Agent 流水线一致。

1. **需求理解 Agent**  
输入：PRD、需求说明、业务背景。  
输出：功能模块、用户角色、关键任务、业务目标、评估边界。  
价值：让评估不只看界面，而是回到“需求希望用户完成什么”。

2. **原则建模 Agent**  
输入：启发式评估原则、问题记录清单。  
输出：系统状态可见性、匹配用户心智、一致性、错误预防、帮助说明、视觉层级等评估维度。  
价值：把抽象原则转成可执行检查点，避免评估口径发散。

3. **旅程拆解 Agent**  
输入：用户画像、PRD 功能、线上页面结构。  
输出：用户体验旅程地图、旅程阶段表、核心任务链路。  
价值：从“页面检查”升级为“任务体验检查”。

4. **任务生成 Agent**  
输入：旅程地图和启发式原则。  
输出：体验导向任务清单、简洁执行版任务清单。  
价值：完整方案给专家把控，简洁版给中低阶设计师顺序执行。

5. **证据采集 Agent**  
输入：线上截图、浏览器采集、问题流程。  
输出：问题界面截图、流程证据、页面上下文。  
价值：让体验问题从主观描述变成有图有路径的证据。

6. **问题归因与报告 Agent**  
输入：任务走查结果、截图证据、原则映射。  
输出：问题描述、启发式原则、严重等级、解决方案、Excel/Markdown 报告。  
价值：让问题可评审、可分派、可跟踪、可复盘。

关键句：

**UXEval Agent 把一次体验评估拆成“理解、建模、执行、取证、归因、沉淀”六个稳定环节，让评估从个人能力变成组织流程。**

**四、03 人机协同模式**

标题：

**03 人机协同：AI 做结构化，设计师做判断**

这里要明确设计师职责边界，不要让人觉得 AI 自动评估就够了。

AI Agent 负责：

- 快速读取 PRD 和模板。
- 整理模块、角色、流程、页面和截图。
- 生成用户旅程和任务清单。
- 将问题映射到启发式原则。
- 按固定格式输出报告。
- 维护截图证据和本地文件产物。

高级体验设计师负责：

- 判断评估目标和业务优先级。
- 判断用户画像是否准确。
- 校准任务清单是否偏功能测试。
- 判断问题是否真正影响体验。
- 判断严重等级和改进建议是否合理。
- 决定最终对外表达和推动策略。

协同机制：

1. **分步确认**  
每一步产物先确认再进入下一步，例如先确认旅程地图，再拆任务，再输出问题清单。

2. **人工校准**  
当任务偏功能测试、内容过简或方向过泛时，由设计师纠偏，让 Agent 回到体验价值判断。

3. **证据闭环**  
每条问题必须有截图或流程证据，不能只输出抽象观点。

关键句：

**AI 承担高频、结构化、可模板化的工作；设计师保留判断、取舍、价值定义和最终质量责任。**

**五、04 对原有设计流程的变革**

标题：

**04 流程变革：体验评估从“后置检查”变成“质量治理”**

这一部分是你最需要强调的。

可以做成 Before / After 对比。

**传统体验评估流程**

1. 需求完成后设计师人工阅读 PRD。  
2. 根据经验列出评估点。  
3. 打开页面逐个走查。  
4. 手工记录问题。  
5. 评审会上解释问题原因。  
6. 输出问题列表，后续跟进依赖个人推动。

问题：

- 起点晚。
- 粒度不稳定。
- 证据弱。
- 复用难。
- 对业务价值表达不足。

**UXEval Agent 流程**

1. PRD、原则、截图进入统一工作区。  
2. Agent 先结构化需求和评估原则。  
3. 设计师确认用户画像和体验旅程。  
4. Agent 生成体验导向任务清单。  
5. 设计师按任务走查并校准问题价值。  
6. Agent 整理截图证据、原则依据和解决方案。  
7. 形成可复用评估报告、执行清单和方法模板。

变化：

- **从页面导向到旅程导向**：不再只看一个页面，而是看用户目标是否能顺畅达成。
- **从经验导向到原则导向**：问题不只靠个人感觉，而是映射到启发式原则。
- **从描述导向到证据导向**：问题必须绑定截图和流程。
- **从一次性交付到资产沉淀**：每次评估都反哺模板、清单和工作流。
- **从单人评审到团队赋能**：高级设计师沉淀方法，中低阶设计师可以按流程执行。

关键句：

**体验评估 Agent 让设计评审不再只是“发现问题”，而是建立一套可以持续运行的体验质量治理流程。**

**八、07 专业知识库与方法论沉淀**

**07 方法沉淀：把体验评估经验转成 Agent 知识库**
