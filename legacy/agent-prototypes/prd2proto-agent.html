02

工作方式转变：从"人工串联"到"人机协作"

把零散的手工操作变成结构化的 Agent 工作流，每个步骤有明确输入、输出和确认点。

原来怎么做的

需求理解偏差、设计质量看个人、原型制作费时间、评审反复改、设计和开发各说各话。根本问题是缺少标准化的流程和工具。

现在怎么做的

PRD 进来 → Agent 解析需求 → 生成设计策略 → 映射组件 → 输出代码 → 质量检查 → 人工确认。6 个 Agent 按顺序执行，每步等你确认后再继续。

变化在哪

从"一个人从头到尾手工做"变成"Agent 做执行，人做决策"。Agent 负责重复劳动，人负责判断方向和把控质量。

人机协同机制 每步等待用户确认，支持中途补充信息，确保 Agent 不失控。

用户确认 → 继续

Agent 生成阶段性产物后，等待用户确认。确认通过则进入下一步，形成正向闭环。

用户修改 → 重生成

用户对产物不满意可提出修改意见，Agent 基于反馈重新生成，直到满足要求。

用户补充 → 增量更新

用户可中途补充信息（如设计草图、竞品参考），Agent 增量更新产物。

问题 2：方法论执行不准 Agent 经常"自由发挥"，不按要求的方法论执行，输出质量参差。
解决：知识库结构化 将方法论拆分为独立 Markdown，每次读取完整上下文，强制 Agent 对照执行。
问题 3：样式与组件混乱 Agent 自己写样式，导致与设计系统不一致，代码无法复用。

踩坑 1

一开始想让 Agent 一次性生成完整设计文档，结果用户根本看不完。后来拆分为 6 个 Agent，每步只问 2-3 个问题，用户确认后才继续。

踩坑 2

Agent 经常"自由发挥"，不按要求的方法论执行。后来把方法论写成结构化 Markdown，每次读取完整上下文，强制对照执行。

踩坑 3

Agent 自己写样式，导致与设计系统不一致。后来强制使用本地组件库，样式变量必须从 foundation.md 提取，不得硬编码。

核心经验 三个关键认知转变，从"理想化"到"可落地"。

经验 1：渐进式披露

不要一次性给太多信息，用户会 overwhelm。采用"渐进式披露"，每步只展示当前需要的内容。

经验 2：约束即自由

给 Agent 的约束越多（模板、示例、检查清单），输出质量越稳定。自由发挥 = 不可控。

经验 3：人机协同

Agent 不是替代人类，而是放大人类能力。关键决策点必须留给人，Agent 负责执行和初稿。

最终目标：让 AI 带来的产出速度，真正转化为可控、可审计、可持续的集团级设计交付能力。 机制关键词：体验先行、契约先稳、工程收敛、质量放行、Agent 共创与愈合。
