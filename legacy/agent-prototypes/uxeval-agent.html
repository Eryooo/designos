01

项目背景：传统体验评估为什么难以规模化？

复杂 B 端产品的体验评估往往依赖资深设计师经验，问题发现质量不稳定、证据不足、复用困难。

传统方式痛点

  • 依赖专家：评估质量高度依赖资深设计师个人经验。
  • 口径发散：有人偏功能测试，有人偏视觉问题，粒度不一致。
  • 需求脱节：评估时容易只看页面，缺少对用户目标和任务路径的校准。
  • 证据不足：问题常停留在“感觉不顺”，难以推动产品和研发认同。
  • 复用困难：项目结束留下问题列表，却没有形成下一次可执行的方法。
  • 价值后置：设计师常在页面完成后介入，影响力集中在发现问题而非质量治理。

UXEval Agent 解决方案

  • 原则驱动：将启发式原则转成稳定的评估维度和检查点。
  • 旅程导向：先建立用户画像和体验旅程，再拆解评估任务。
  • 证据闭环：每条问题绑定界面截图、流程上下文和判断依据。
  • 人类校准：AI 负责结构化，设计师负责价值判断和质量放行。
  • 产物复用：输出完整方案、简洁执行版、问题模板和采集脚本。
  • 组织赋能：把高级设计师经验转化为团队可复制的体验质量基础设施。
02

UXEval Agent 协同工作流

把一次体验评估拆成“理解、建模、执行、取证、归因、沉淀”六个稳定环节。

1

需求理解

需求理解 Agent
读取 PRD、业务背景和评估目标，抽取功能模块、用户角色、关键任务和评估边界。

2

原则建模

原则映射 Agent
将启发式原则转为系统状态、用户心智、一致性、错误预防、帮助说明、视觉层级等维度。

3

旅程拆解

旅程建模 Agent
基于用户画像和线上页面结构，生成用户体验旅程地图、旅程阶段和核心任务链路。

4

任务生成

任务生成 Agent
输出体验导向任务清单和设计师简洁执行版，让专家把控与团队执行分层。

5

证据采集

证据采集 Agent
结合线上截图、浏览器采集和问题流程，沉淀可追踪的问题界面证据。

6

报告输出

问题归因 Agent
输出问题描述、启发式原则、严重等级、建议方案和 Markdown / Excel 报告。

03

人机协同：AI 做结构化，设计师做判断

AI 提升评估效率，但体验价值、问题优先级和最终质量责任仍由设计师掌控。

AI Agent 负责

读取资料、整理页面、生成旅程、拆解任务、映射原则、整理截图、结构化输出报告。

高级设计师负责

定义评估目标、校准用户画像、判断体验影响、确定严重等级、把控建议质量和推动策略。

协同工作方式

每个阶段形成产物后由设计师确认，发现方向偏差及时纠偏，确保 Agent 不滑向纯功能检查。

协同工作三大原则 让 AI 的结构化能力与设计师的专业判断形成互补。

分步确认

先确认旅程地图，再确认任务清单，最后执行问题输出，避免错误判断在后续环节累积。

人工校准

当任务过度偏功能测试或表达过泛时，由设计师纠偏，回到体验流程、操作感受和业务价值。

证据闭环

每条体验问题必须有流程或界面截图，避免只有观点没有证据，提升跨角色沟通效率。

04

06

专业知识库：把体验评估经验转成 Agent 资产

高级设计师的隐性经验被拆解为原则库、旅程模板、任务模板、问题模板、证据规范和报告规范。

启发式原则库

系统状态、用户心智、一致性、错误预防、帮助说明、视觉层级、数据可信等维度。

用户旅程模板

进入与定位、前置配置、任务执行、结果理解、报告决策、闭环优化等阶段。

体验任务模板

统一沉淀任务、场景、子任务、评估重点、启发式原则和证据记录要求。

问题记录模板

问题流程、截图证据、问题描述、原则依据、严重等级、建议方案和处理状态。

证据采集规范

截图命名、流程记录、登录信息本地化、敏感信息保护和报告引用方式。

报告输出规范

Markdown 便于评审阅读，Excel 便于分派跟踪，HTML 便于分享传播。

核心经验总结

1
把经验变成结构，AI 才能稳定执行先把原则、任务、证据和报告格式显性化，再让 Agent 执行。
2
把流程拆成阶段，设计师才能持续校准每个阶段都有确认点，避免一次性生成带来方向偏差。
3
把问题绑定证据，体验建议才更容易推动截图、流程、原则和建议共同构成跨角色沟通语言。
4
把产物做成模板，个人能力才能变成团队能力完整方案给专家，简洁版给执行者，报告模板给协作闭环。
最终目标：把高级体验设计师的评估能力封装成团队可复用的体验质量基础设施。 核心关键词：需求理解 · 旅程建模 · 原则映射 · 证据化问题 · 人类校准 · 组织复用。
