# 证据采集方法论

## 定义

证据（Evidence）是「锚定一条 Issue 到具体页面 / DOM / 流程的物证」。
没有证据的 Issue 在评审会上无法推动，违反 UXEval 宪法 #1。

## 证据类型

| Kind | 适用 | 优先级 |
|---|---|---|
| `screenshot` | 单页面问题 | 高（默认） |
| `dom` | 结构性问题（如 Tab 顺序、对比度） | 高 |
| `trace` | 流程性问题（多页面跳转、长操作） | 中 |
| `video` | 动画 / 时序问题 | 低（仅在静态截图无法表达时） |

## 截图规范

### 命名约定

这是**推荐的自动化加速项**，不是 client 模式的准入门槛。
如果截图很多，系统会先自动起草 screenshot -> page/state 映射。优先补 `screens-description.md`，只有少数歧义截图再补 `screens-map.md` 或 `screens-index.md`，不要先花大量时间批量重命名。

```
{问题ID}-{简述}-{状态}.png
```

例：
- `I-001-工作台-默认.png`
- `I-001-工作台-悬停.png`
- `I-005-规则编辑-错误状态.png`

### 必须包含的状态（关键交互组件）

针对交互组件（按钮、表单、下拉），关键状态都要截：
- 默认（default）
- 悬停（hover）
- 按下（active）
- 聚焦（focus）
- 禁用（disabled）
- 加载中（loading）
- 错误（error）

不是每个问题都要 7 张图，但「按钮无 hover 反馈」这类问题必须默认 + 悬停两张对比。

### bbox 标注

每张截图如果指向具体元素，必须在 `Evidence.bbox` 中给出 `(x, y, w, h)`。
让生成 Excel 报告时可以加红框。

## Web 模式：Playwright 脚本生成规则

### 脚本结构

每条体验任务（task）生成一个 `eval_*.spec.mjs` 脚本：

```javascript
// eval_T-005-rule-draft-recovery.spec.mjs
import { test, expect } from '@playwright/test';

test.describe('T-005 规则草稿保存与恢复', () => {
  test.use({ storageState: process.env.APP_AUTH_STATE_PATH });
  
  test('编辑后关闭重开是否保留内容', async ({ page, context }) => {
    // Step 1: 进入规则编辑
    await page.goto(`${process.env.APP_BASE_URL}/rules/new`);
    await page.screenshot({ path: 'evidence/T-005-01-default.png', fullPage: true });
    
    // Step 2: 填写部分字段
    await page.fill('#rule-name', '测试规则_uxeval');
    await page.fill('#rule-desc', '描述内容');
    await page.screenshot({ path: 'evidence/T-005-02-filled.png', fullPage: true });
    
    // Step 3: 模拟用户关闭
    await page.close();
    
    // Step 4: 新页面重新打开
    const page2 = await context.newPage();
    await page2.goto(`${process.env.APP_BASE_URL}/rules/new`);
    await page2.screenshot({ path: 'evidence/T-005-03-reopen.png', fullPage: true });
    
    // 采集 DOM 状态用于后续分析
    const ruleNameValue = await page2.locator('#rule-name').inputValue();
    const result = {
      task_id: 'T-005',
      content_preserved: ruleNameValue === '测试规则_uxeval',
      reopen_value: ruleNameValue,
    };
    await page2.evaluate((data) => {
      window.__uxeval_result = data;
    }, result);
  });
});
```

### 脚本生成约束

- **不写断言**：脚本只采集，不判定（判定交给 heuristic-engine）
- **每步都截图**：每个 step 至少 1 张图
- **必须复用登录态**：使用 `storageState` 而非每次登录（避免账号被锁）
- **失败不抛错**：用 `try/catch` 包裹关键步骤，失败也要记录截图，但要在 `result` 里标记 `step_failed: true`
- **超时设置**：默认 30s/step，关键操作 60s
- **不修改业务数据**：尽量用「测试 + 时间戳」前缀的数据，跑完清理

### 反爬 / 被识别成机器人

- 使用真实 User-Agent
- 加入随机 sleep（500ms-1500ms）
- 不要用 headless 模式做敏感操作（部分系统会检测）

## Client 模式：截图规范

### 用户提供的截图必须

- 文件名遵循「{模块}-{页面}-{状态}.png」
- 关键流程必须有「初始 → 中间 → 结果」三连
- 截图前关闭无关插件、广告，避免噪音

### Agent 自动校验

`screenshot-loading` 阶段会用 `image-analyzer` 检查：
- `required_evidence_plan`：先规划本次 run 最关键需要哪些页面、哪些状态、哪些说明
- `evidence_input_guidance`：开跑前一次性说明当前输入缺什么、该怎么补
- 递归发现截图与 `.md` 说明文件
- 单图分辨率与文件大小
- 截图可读性判断
- `.md` 说明文件 preview 与截图关联
- 本地 OCR 可用时提取 OCR 文本、按钮 / 导航 / 状态词
- 基于文件名 / OCR / `.md` 文本的风险提示
- `evidence_assessment`：同时判断当前输入是否足够继续，以及是否达到最终交付标准

当前**不检查**：
- 像素级敏感信息
- 完整页面语义总结
- task / module 自动匹配
- 场景意图自动判断

如果满足以下任一情况，`evidence_assessment` 应为 `supplement_needed` 或 `blocked`：
- 截图分辨率过低
- 本地 OCR 不可用且没有说明文件
- OCR / markdown / 文件名三路线索都不足
- 在无 OCR、无 markdown、也无轻量映射时，关键截图命名混乱进一步放大了匹配风险
- 对照 `required_evidence_plan` 后，关键页面 / 关键状态 / 关键说明仍明显缺失

这时不要继续交付“看起来完整”的问题清单，优先要求补：
- `required_evidence_plan` 指定的关键页面截图
- `required_evidence_plan` 指定的关键状态截图
- `inputs/screens-description.md`
- `inputs/screens/screens-map.md` 或 `inputs/screens/screens-index.md`
- 高分辨率关键截图
- 页面流程说明
- 如你方便，再小范围补关键截图命名

此外还要区分最终交付资格：
- `delivery_status = final_delivery_ready`
  - 证据质量足够支持最终问题清单和最终报告
  - 目标质量接近 99%-100%
- `delivery_status = fallback_safe`
  - 证据只够输出受限中间结果
  - 安全质量目标至少 85%
  - 不能直接交付最终 issue_report / html_report
  - 后续必须经过 deterministic delivery-audit，自动生成 bounded fallback package
- `delivery_status = supplement_required`
  - 证据不足以形成 issue 主清单，必须先补资料
- `delivery_status = blocked`
  - 证据完全无法继续

## 证据组织

### 目录结构

```
runs/001-uxeval/evidence/
├── screens/
│   ├── T-005-01-default.png
│   ├── T-005-02-filled.png
│   └── T-005-03-reopen.png
├── dom/
│   └── T-005-dom-snapshot.json
├── traces/
│   └── T-005-trace.zip
└── manifest.yaml          # 证据清单：每张图对应哪个 task / issue
```

### manifest.yaml 格式

```yaml
evidences:
  - id: E-001
    kind: screenshot
    path: screens/T-005-01-default.png
    bbox: [120, 240, 800, 400]
    related_task: T-005
    related_issue: I-005
    note: 编辑器初始空白状态
  - id: E-002
    kind: dom
    path: dom/T-005-dom-snapshot.json
    related_task: T-005
    note: 关闭前 DOM 快照
```

## 注意事项

- **截图必须是源文件**，不要截图后再截图
- **DOM 快照只采业务相关节点**，不要全文档（动辄 10MB+）
- **trace 文件保留 7 天后自动归档**，避免膨胀
- **敏感信息打码**：身份证、手机号、客户名、内部域名前缀
