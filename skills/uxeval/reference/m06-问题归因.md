# 问题归因方法论

## 定义

问题归因（Issue Attribution）= 把 heuristic-engine 输出的 raw issue（含问题描述 + 触发原则 + 截图引用）转成结构化 Issue（含归因到旅程阶段 + 任务 + 模块 + 严重等级 + 可执行建议）。

输出 Issue 必须严格符合 `kernel/contracts/schemas.py:Issue` 定义。

## 步骤

### Step 1：归因到旅程阶段（journey_stage_id）

每个 raw_issue 必须指向一个 `JourneyStage`。
根据问题发生的页面 / 操作，匹配旅程地图。

匹配规则（优先级从高到低）：
1. 截图显式标注的页面属于哪个 stage
2. 触发原则的 `applicable_modules` 与某个 stage 的关联模块匹配
3. 问题文本中明确提到的操作（「保存」「提交」）属于哪个 stage

无法归因到任何 stage 的 issue：
- 标记 `journey_stage_id: null`
- 在 `description` 中加 `[未归因]` 前缀
- 数量超过 issues 总数 30% 时报警（说明旅程地图不够覆盖）

### Step 2：归因到任务（task_id）

可选字段，能标尽量标。
通常一条 issue 在执行某个 task 时被发现，task_id 直接复用。

### Step 3：归因到模块（module_id）

必填。每个 issue 必须落到一个 Module。
如果 issue 跨多 Module（如「全局导航」），创建一个虚拟 Module `M-GLOBAL`。

### Step 4：判定严重等级

按下表决策（不可拍脑袋）：

| 用户影响 | 频次 | 严重等级 |
|---|---|---|
| 无法完成核心任务 | 任意 | `critical` |
| 数据 / 资金 / 合规风险 | 任意 | `critical` |
| 显著阻碍效率（多步绕过、等待 > 10s） | 高频（每天遇到） | `major` |
| 显著阻碍效率 | 低频（偶尔） | `minor` |
| 体验不佳但可完成 | 任意 | `minor` |
| 优化建议（不影响行为） | 任意 | `suggestion` |

### Step 5：写用户影响（user_impact）

强制结构：
```
在 {场景} 下，{角色} {做什么} 时，遇到 {问题现象}，导致 {影响}
```

四要素缺一不可，违反宪法 #6。

### Step 6：写改进建议（suggestion）

强制三要素（违反宪法 #5）：

1. **改什么**：具体元素 / 流程 / 文案
2. **改成什么**：具体方向（不必给设计稿）
3. **为什么**：链接到原则 + 用户影响

模板：
```
建议把 {元素} 从 {现状} 改为 {目标}，
因为当前违反 {原则 ID}（{原则名}），
{用户实际影响数据 / 推断}。
```

### Step 7：标基准来源（source_basis）

| 当 PRD 与实现冲突时 | source_basis |
|---|---|
| 实现没满足 PRD 要求 | `prd` |
| PRD 没写、实现自由发挥 | `screenshot` |
| 都没明确，但违反通用启发式 | `inferred` |

### Step 8：组装 Issue

```yaml
- id: I-001
  title: 工作台规则待办与其他待办无视觉区分
  description: 在「早晨查看待办」场景下，运营专员进入工作台，需要从 12 条混合待办中找出规则相关，但所有待办字号、颜色、图标都相同，平均需 8 秒才能定位 5 条规则待办。
  severity: major
  principle_ids: [S2, P3]
  journey_stage_id: JS-001
  task_id: T-001
  module_id: M-WORKBENCH
  evidence_refs: [E-001, E-002]
  user_impact: |
    每天早晨运营专员需 8-15 秒定位规则待办（实测平均 11s），
    且漏看率 23%（5 名用户测试 20 次，4 次漏看至少 1 条）。
    高频场景下累计认知负担显著，影响工作启动效率。
  suggestion: |
    建议为规则相关待办增加视觉标识（建议使用「规则」标签 + 蓝色侧边条），
    与其他类型待办（如审批、消息）形成视觉层级。
    因为当前违反 S2（再认而非记忆）：用户必须读完文本才能识别类型；
    以及 P3（一致性与标准化）：所有待办视觉权重相同，无法快速扫描。
  source_basis: screenshot
```

## 严重等级判定参考

### critical 标准（任一满足即可）

- 用户无法完成 KeyTask 中至少一步
- 数据丢失 / 错误数据被使用 / 误删除不可恢复
- 资金 / 合规风险（如错误的费率、违反法规）
- 50%+ 用户在该步骤失败

### major 标准

- 用户能完成但需绕路（额外 3+ 步骤 / 等待 > 10s）
- 高频场景（每天 / 每次都遇到）
- 跨模块一致性问题（多页面违反）

### minor 标准

- 体验不佳但不影响完成
- 低频场景
- 单页面孤立问题

### suggestion 标准

- 不影响用户行为，仅是优化方向
- 涉及主观偏好（视觉风格）
- 长期改进建议（如重构信息架构）

## 去重规则

raw_issues 通常有重复，必须去重：
- 同一原则在同一 module 触发多次 → 合并为 1 条 issue，evidence_refs 累加
- 跨多 module 的同类问题（如「全产品按钮一致性」）→ 合并到虚拟 module M-GLOBAL
- 已知问题（在 `inputs/historical-issues.md` 中）→ 标记 `is_known: true`，不进入主报告

## 输出 issues 列表的总体约束

- 一次评估输出 issues 数量建议 15-40 条
- < 10 条：评估深度不够，回退到 task-generation 检查任务覆盖
- > 50 条：粒度太细，按上面去重规则合并
- critical 占比 > 30%：怀疑标过严，请人工 calibrate
- 没有任何 critical：怀疑没真测试关键路径

## 注意事项

- ❌ 不要把功能存在与否写成 issue（违反宪法 #4）
- ❌ 不要写「按钮丑」这种无影响描述（违反宪法 #6）
- ❌ 不要让 evidence_refs 为空（违反宪法 #1）
- ✅ 每条 issue 都要过一遍 7 条宪法
- ✅ 标 source_basis 是必填，不许偷懒（违反宪法 #7）
