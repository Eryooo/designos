# 需求理解方法论

## 定义

把一份非结构化的 PRD（往往是 PDF / docx / 飞书文档）转成评估可消费的 5 类结构化数据：

1. **Module**：功能模块（如「数据规则配置」「分类管理」）
2. **UserRole**：使用产品的用户角色（如「运营专员」「数据管理员」）
3. **Scenario**：高频场景（如「新增分类」「批量调整规则」）
4. **KeyTask**：关键任务链路（跨多个模块的端到端任务）
5. **EvaluationBoundary**：评估边界（哪些模块在范围内，哪些不在）

## 步骤

### Step 1：识别业务目标

读 PRD 前 3 节（通常是「项目背景」「业务目标」「目标用户」），提炼一句话目标。
**模板**：「为 {角色} 提供 {能力}，达成 {业务结果}」

### Step 2：拆模块

按 PRD 章节结构 + 功能分组提取，每个 Module 必备：
- `id`：M-001 起编号
- `name`：8 字以内
- `description`：一句话说明边界
- `pages`：明显归属的页面 ID（无需穷举所有 URL，关键页面即可）

合并规则：
- 同一业务流的多个子页面合并为 1 个 Module
- 「设置 / 帮助 / 关于」这类辅助模块在 client 模式下默认排除

### Step 3：推导角色

从 PRD 的「目标用户」「权限设计」「角色矩阵」三处提取，每个 Role 必备：
- `id`：R-001 起编号
- `name`：职业岗位（不是「张三」）
- `goals`：使用产品的 3-5 个目标
- `pain_points`：现有方案的 2-4 个痛点（PRD 没明说时由 Agent 推断，标记 `[inferred]`）

去重规则：
- 「数据管理员」和「数据运维」在 PRD 里通常是同一角色
- 「超级管理员」如果只是权限差异不是 UX 角色，合并到对应业务角色

### Step 4：推导场景

每个 Module 至少 1 个高频场景，每个高频场景必备：
- 触发条件（用户什么时候做这件事）
- 跨模块路径（涉及哪些 Module）
- 期望结果

### Step 5：识别关键任务

跨多模块的、高频、影响业务核心指标的任务。
通常一个产品有 3-7 个 KeyTask（少了说明拆得太细，多了说明没收敛核心）。

### Step 6：划评估边界

明确写出 `out_of_scope` 列表，至少包含：
- 暂未上线的功能
- 第三方嵌入页面
- 一次性引导（如初始化向导）

## 模板

```yaml
modules:
  - id: M-001
    name: 数据规则配置
    description: 创建、调整、版本化数据分类规则
    pages: [/rules, /rules/edit, /rules/version]
roles:
  - id: R-001
    name: 运营专员
    goals:
      - 快速创建并调整分类规则
      - 复用已有规则模板
    pain_points:
      - 规则版本切换会误覆盖在用规则
      - 批量调整缺少撤销
scenarios:
  - id: S-001
    role_id: R-001
    name: 新增分类规则
    trigger: 业务方提出新分类口径
    cross_modules: [M-001, M-003]
    expected_result: 规则上线 + 历史数据回溯成功
key_tasks:
  - id: KT-001
    name: 端到端规则上线
    description: 从规则草稿到全量数据生效
    involved_modules: [M-001, M-002, M-003]
    involved_roles: [R-001, R-002]
evaluation_boundary:
  in_scope: [M-001, M-002, M-003]
  out_of_scope:
    - 用户登录与权限管理（独立 SSO 系统）
    - 系统设置（非业务核心）
```

## 示例

**PRD 节选**：
> 「数据分类分级规则配置平台」面向数据治理团队，
> 由运营专员配置规则，数据管理员审批，最终由系统应用到全量数据。
> v1.0 范围：规则草稿、版本管理、审批流；
> v1.1 计划：规则市场、AI 推荐（不在本次评估）。

**结构化结果**：
- Modules: 规则草稿 / 版本管理 / 审批流
- Roles: 运营专员（创建）/ 数据管理员（审批）
- KeyTask: 规则草稿 → 提交审批 → 生效
- 评估边界：v1.0 三个模块 in_scope；规则市场、AI 推荐 out_of_scope

## 注意事项

- **不要替 PRD 补功能**：PRD 没写的「期望功能」不要列为 Module，会污染后续阶段
- **角色不要堆砌**：3-6 个角色最合理，超过 8 个一定是粒度太细
- **`[inferred]` 标记必须保留**：让设计师知道哪些是 Agent 推断、哪些是 PRD 原文
- **不要在这一步判断好坏**：需求理解只做提取，价值判断留给后续 stage
