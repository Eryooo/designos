# 任务生成方法论

## 定义

**体验任务**（Experience Task）= 一个可被设计师/AI 执行的、能产出体验问题的最小评估单元。

任务清单分两套：
- **完整版**（任务清单-完整版.md）：含场景、子任务、评估重点、证据要求，给资深设计师做整体把控
- **简洁执行版**（任务清单-简洁版.md）：5 字段精简，给中低阶设计师 / 自动化脚本顺序执行

## 业务文档示例（合成）

> 以下为**合成示例**，演示文档结构，不含任何真实产品/业务证据。

### 示例 1：用户旅程地图（合成：某 B 端教学工具）

真实业务中的旅程地图包含以下结构：

```yaml
journey_map:
  - stage_name: 登录
    stage_tasks:
      - task: 登录网站
        user_behaviors: [账号密码登录平台]
        mot_goals: [方便快捷的登录]
  
  - stage_name: 作业设计
    stage_tasks:
      - task: 切换身份
        user_behaviors: [找到并进入目标工具模块]
        mot_goals: [入口清晰]
      
      - task: 进入作业设计
        user_behaviors: [进入作业设计，并选择作业类型]
        mot_goals: [入口清晰，方便选择]
      
      - task: 选择教材确认章节
        user_behaviors: [选择教材版本及章节目录]
        mot_goals: [提示清晰，避免用户出错]
      
      - task: 试题加篮
        user_behaviors: [浏览题库，并点击加入试题篮]
        mot_goals: [题库丰富，筛选过程简单高效]
  
  - stage_name: 试题编辑
    stage_tasks:
      - task: 选题预览
        user_behaviors: [进入试题预览页面]
        mot_goals: [入口清晰，方便选择]
      
      - task: 增删调整
        user_behaviors: [试题预览页面完成试题增删、排序等操作]
        mot_goals: [增删、排序操作直观清晰，方便操作]
  
  - stage_name: 题卡编辑
    stage_tasks:
      - task: 题卡编辑
        user_behaviors: [进入题卡制作，完成题卡样式排版]
        mot_goals: [满足教学实际场景，易学习]
      
      - task: 布置作业
        user_behaviors: [点击布置作业，在布置作业弹窗选择对应配置]
        mot_goals: [满足教学实际场景，易学习]
```

**关键要素**：
- **阶段（stage）**：用户完成目标的关键阶段
- **阶段任务（stage_tasks）**：每个阶段的具体任务
- **用户行为（user_behaviors）**：用户在该任务中的具体操作
- **MOT 目标（mot_goals）**：关键体验时刻（Moment of Truth）的目标

### 示例 2：可用性测试任务脚本（合成：某 B 端教学工具）

合成示例的任务脚本格式：

```yaml
usability_test_script:
  - scenario: 作业设计
    estimated_time: 15min
    intro: |
      老师好，非常感谢你能抽空参加此次体验活动，
      接下来请您按照以下指引使用该工具完成一次学生作业布置，
      过程中有任何疑问和困惑都可以向我们反馈，感谢您的配合。
    
    steps:
      - name: 登录网站
        script: |
          进入平台官方网站，登录账号；
          账号/密码：<test_account>/<test_password>
          或者教师自有账号
        completion: S
      
      - name: 切换身份
        script: 切换登录账号身份为教师账号
        completion: S
      
      - name: 进入作业设计
        script: |
          该工具提供了丰富的题库资源，题目来源包括同步作业、专项作业等；
          您今天完成了某章节、某课时的授课内容，
          现在需要使用该工具布置一套课后作业练习题；
          任务1：切换资源-切换教材资源为【某教材版本】
          任务2：词汇练习-进入词汇默写卡，设计一份某单元的词汇练习（题目控制在10道左右）
          任务3：课时作业-进入同步课时练，请设计一份某单元某模块的课时作业，题目数量控制在20题左右
          任务4：综合测试-设计一份某单元的单元检测卷，需包含「词汇题、填空题」「作文题」题目控制在10道左右
        completion: S
      
      - name: 选题预览
        script: |
          挑选完题目后，您可以进入到预览界面，整体预览所有题目，
          在预览界面中您可以继续进行题目新增、删除、排序的操作；
          现在需要您对试卷进行调整，要求如下：
          任务1：请您将题目按照题型模块排序；
          任务2：批量删除第一道、第三道题；
          任务3：请您任意调整两个题目间的顺序；
          完成以上操作后，点击去制卡完成作业布置
        completion: S
      
      - name: 题卡编辑
        script: |
          题目确认后，您还需要对试卷的样式进行一些调整，
          可以调整试卷的字体大小、行高行距、试卷尺寸等设置项，
          除此之外您还可以对题目进行再次编辑，请按照要求进行题卡编辑，要求如下：
          任务一：题卡相关
          任务1：调整作业名称"某班级某科目测验"
          任务2：检查作业分数以保持通过
          任务3：适当调整题面高度、行高、行距、字体大小等，使纸张面积可以最大化被利用，并把作业下载到本地
          任务二：词汇相关
          任务1：调整词汇题型空白作答区域的长短
          任务2：调整纸张设置成A4尺寸、双栏
        completion: S
  
  - scenario: 作业扫描
    estimated_time: 10min
    steps:
      - name: 开机放纸
        script: |
          作业布置后，学生完成了作业作答与回收，您可以使用智能扫描仪完成扫描、智能批改。
          请按照要求完成作业扫描，要求如下：
          任务1：将扫描仪开机，并检查网络连接状态，设置学科、班级信息
          任务2：将作业卡放置在扫描仪进纸口，点击开始扫描
          任务3：查看没有匹配到学生信息的作业卡
          任务4：作业卡扫描完成退出扫描
        completion: S
      
      - name: 扫描纠错
        script: 检查扫描过程中的错误并进行纠正
        completion: P
  
  - scenario: 报告查看
    estimated_time: 8min
    steps:
      - name: 作业查找
        script: |
          作业完成扫描后，您可进入作业列表，预览过往所有布置的作业，
          并在作业列表中可对作业预览、报告查看、下载题卡等，
          请按照要求完成作业查找与报告查看，要求如下：
          任务1：请您筛选一份上周布置的作业
          任务2：下载该份作业的题卡
          任务3：找到上周布置的作业，并复制这份作业
          任务4：找到一份上周扫描后的作业，点击查看该份作业报告内容
        completion: S
```

**关键要素**：
- **scenario**：场景名称
- **estimated_time**：预计时长
- **intro**：引导语（可选）
- **steps**：步骤列表
  - **name**：步骤名称
  - **script**：引导语（告诉执行人具体怎么做）
  - **completion**：完成判定（S 成功 / P 部分完成 / F 失败）

### 示例 3：基于旅程地图的任务执行清单

基于上述旅程地图，生成的任务执行清单应该包含：

```yaml
task_checklist_full:
  - task_id: T1
    journey_stage_id: S1  # 对应"登录"阶段
    task_name: 快速登录体验
    evaluation_focus:
      - 用户是否能快速找到登录入口
      - 用户是否理解登录方式（账号密码 vs 其他方式）
      - 用户是否敢点击登录按钮（是否担心出错）
    evidence_requirements:
      - 截图：登录页面、输入账号密码后的状态、登录成功页面
      - 记录：用户是否犹豫、是否尝试其他登录方式、登录耗时
    prd_screenshot_conflict_points:
      - PRD 说支持多种登录方式，截图只有账号密码登录（标注"需补充现场验证"）
    completion_criteria:
      - S：用户一次性完成登录，无挫折，耗时 < 30s
      - P：用户完成登录，但有犹豫或尝试其他方式
      - F：用户无法完成登录或耗时 > 2min
  
  - task_id: T2
    journey_stage_id: S2  # 对应"作业设计-进入作业设计"阶段
    task_name: 作业设计入口识别
    evaluation_focus:
      - 用户是否能快速找到作业设计入口
      - 用户是否理解不同作业类型的区别
      - 用户是否能预测点击后会发生什么
    evidence_requirements:
      - 截图：首页、作业设计入口、作业类型选择页面
      - 记录：用户查找入口的路径、是否犹豫、是否点错
    prd_screenshot_conflict_points:
      - PRD 说有"快速作业"和"自定义作业"两种入口，截图只有一个入口（标注"需补充现场验证"）
    completion_criteria:
      - S：用户 5s 内找到入口，无犹豫
      - P：用户找到入口，但有犹豫或尝试其他路径
      - F：用户无法找到入口或耗时 > 1min
  
  - task_id: T3
    journey_stage_id: S2  # 对应"作业设计-试题加篮"阶段
    task_name: 题库筛选与加篮效率
    evaluation_focus:
      - 用户是否能快速理解筛选条件
      - 用户是否能高效找到目标题目
      - 用户是否理解"加入试题篮"的操作和结果
    evidence_requirements:
      - 截图：题库页面、筛选条件、试题篮状态
      - 记录：用户筛选路径、加篮操作次数、是否重复操作
    prd_screenshot_conflict_points:
      - PRD 说支持"按章节"和"按知识点"筛选，截图只有"按章节"（标注"需补充现场验证"）
    completion_criteria:
      - S：用户 2min 内完成 10 道题加篮，无重复操作
      - P：用户完成加篮，但有重复操作或耗时较长
      - F：用户无法完成加篮或频繁出错
```

## 体验任务 vs 功能任务（核心判别）

这是 UXEval 最容易跑偏的地方。

| 维度 | 体验任务（要） | 功能任务（不要） |
|---|---|---|
| 关注点 | 用户能否顺畅完成 | 功能是否存在 / 是否符合 PRD |
| 提问方式 | 「在这一步，用户感受如何」 | 「这个按钮点击是否跳转到 X 页面」 |
| 验收标准 | 体验维度（速度、清晰度、可控） | 功能维度（结果对不对） |
| 归属 Skill | UXEval | design-acceptance / 测试团队 |

### 判别口诀

> 「点不点得了」是功能问题；「点不点得到 / 点了爽不爽」是体验问题。

### 反例

❌ **功能任务（不要）**：「点击保存按钮，验证规则提交成功」
✅ **体验任务（要）**：「在保存规则的过程中，用户能否清楚知道当前进度，遇到错误能否理解并恢复」

❌ **功能任务**：「检查列表页排序是否正确」
✅ **体验任务**：「用户从列表页快速定位目标记录的效率（包含搜索、筛选、排序的体验）」

## 步骤

### Step 1：每个旅程阶段拆 2-5 个任务

不要在一个 `journey_stage` 下塞 10 个任务，会失焦。
原则：每个 pain_hotspot 至少对应 1 个任务。

### Step 2：每个任务必备字段

```yaml
- id: T-001
  title: 在工作台上一眼定位待办规则
  role: 运营专员
  scenario: 早上登录后看当日工作
  journey_stage_id: JS-001
  description: 用户登录后进入工作台首页，需要在 5 秒内定位到当日需要处理的规则任务
  prerequisites:
    - 已登录
    - 有 ≥ 3 条待办
  steps:
    - 1. 进入工作台首页
    - 2. 查看待办区域
    - 3. 识别哪些是规则相关待办
  success_criteria:
    - 5 秒内识别出规则待办数量
    - 能区分「我创建」和「待我审批」
  applicable_principles: [F2, S2, P3]
  evidence_requirements:
    - 工作台首页截图（关键区域有 bbox 标注）
    - 鼠标悬停状态截图
```

### Step 3：标注覆盖度

任务清单整体要保证：
- 每个 Module 至少 2 个任务
- 每条 KeyTask 至少 3 个任务（首尾 + 中间）
- 每条 pain_hotspot 至少 1 个任务

### Step 4：生成简洁版

简洁版字段：

```yaml
- id: T-001
  title: 工作台快速定位待办
  role: 运营专员
  steps_summary: 登录 → 工作台首页 → 识别规则待办
  must_check: F2, S2, P3（系统状态可见性 / 再认而非记忆 / 一致性与标准化）
```

5 字段足以让中低阶设计师按顺序执行。

### Step 5：拒绝功能测试任务

每个任务在生成时必须过一遍判别口诀。
如果发现「这就是测试用例」，删掉或重写为体验视角。

## 示例（部分）

**完整版**：

```yaml
- id: T-005
  title: 规则草稿保存与恢复
  role: 运营专员
  scenario: 编辑规则中途被打断（开会、电话）
  journey_stage_id: JS-003
  description: |
    运营专员在编辑长规则时被中断，重新打开后能否找到上次编辑内容。
    规则编辑表单字段约 15 个，重新输入成本高。
  prerequisites:
    - 进入规则编辑页面
    - 至少填写 5 个字段
  steps:
    - 1. 在规则编辑页填写部分字段
    - 2. 关闭浏览器标签
    - 3. 重新打开同一规则
    - 4. 检查上次内容是否还在
  success_criteria:
    - 上次未提交的内容自动保留
    - 或：明确提示「未保存，是否丢弃」
  applicable_principles: [S1, F1]
  evidence_requirements:
    - 关闭前截图
    - 重新打开后截图
```

**简洁版**：

```yaml
- id: T-005
  title: 草稿保存与恢复
  role: 运营专员
  steps_summary: 编辑 → 关闭 → 重开 → 检查内容
  must_check: S1 用户控制度与自由度（是否丢失数据）, F1 预防出错
```

## 任务规模建议

| 产品规模 | 总任务数 | 简洁版任务数 |
|---|---|---|
| 小型工具（< 5 模块） | 8-15 | 6-10 |
| 中型 B 端（5-10 模块） | 15-30 | 10-20 |
| 大型平台（10+ 模块） | 30-50 | 20-35 |

任务数过 50 一定是粒度太细，请合并。

## 注意事项

### 不要做的事

- ❌ 把 PRD 的「需求点」直接抄成任务（那是 design-acceptance 的活）
- ❌ 给同一个 pain_hotspot 拆 5 个任务（拆 2 个就够）
- ❌ 任务的 success_criteria 写成「按钮可点击」（这是测试，不是体验）
- ❌ 跳过 `applicable_principles` 字段（这是 issue-attribution 的关键）

### 需要做的事

- ✅ 让每个任务都对应至少 1 条原则
- ✅ 简洁版剥离场景细节，但保留 must_check
- ✅ evidence_requirements 写明「截图什么」（让证据采集阶段不用猜）

## 异常场景任务示例

### 示例 1：上下文切换验证

**任务 ID**: T-EX-001  
**任务名称**: 验证多工作空间切换时的状态保持  
**场景**: 用户在空间A编辑规则时切换到空间B，再返回空间A  
**执行步骤**:
1. 在空间A创建新规则，填写部分字段（不保存）
2. 切换到空间B
3. 在空间B执行任意操作
4. 切换回空间A

**成功标准**:
- 空间A的编辑器状态保留
- 已填写的字段内容不丢失
- 页面不出现错误提示

**证据要求**: 截图（切换前、切换后、返回后）

---

### 示例 2：页面渲染容错

**任务 ID**: T-EX-002  
**任务名称**: 验证模型库大数据量渲染  
**场景**: 模型库列表加载100+个模型  
**执行步骤**:
1. 构造100+个模型的测试数据
2. 进入模型库页面
3. 观察页面渲染表现
4. 尝试滚动、搜索、筛选

**成功标准**:
- 页面不出现白屏
- 滚动流畅（FPS > 30）
- 搜索/筛选响应时间 < 1s

**证据要求**: 性能截图 + 控制台日志

---

### 示例 3：网络异常恢复

**任务 ID**: T-EX-003  
**任务名称**: 验证模型训练中断网后的恢复  
**场景**: 模型训练过程中网络中断  
**执行步骤**:
1. 启动模型训练
2. 训练进行到50%时断网（关闭WiFi）
3. 等待30秒
4. 恢复网络

**成功标准**:
- 页面显示"网络中断"提示
- 恢复网络后自动重连
- 训练进度不丢失（或从断点续传）

**证据要求**: 截图（断网提示、恢复后状态）
