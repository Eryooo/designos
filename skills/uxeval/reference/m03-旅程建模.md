# 旅程建模方法论

## 定义

用户体验旅程（User Journey Map）是「以用户视角，串联多个功能页面，描绘完成一个目标的全过程」。
旅程建模解决一个核心痛点：**评估如果按页面/菜单顺序做，会忽略页面之间的衔接体验问题**。

UXEval 用 6 阶段通用模板：

1. **进入与定位**（用户如何进入产品 / 找到自己要做的事）
2. **前置配置**（开始任务前的设置 / 准备）
3. **任务执行**（核心操作链路）
4. **结果理解**（任务后的数据查看 / 结果验证）
5. **报告决策**（基于结果做下一步决策）
6. **闭环优化**（基于决策调整、复盘、复用）

不是每个产品都 6 阶段都有，但建议至少覆盖前 4 个。

## 步骤

### Step 1：选择主角

从 `roles` 列表中挑 1-2 个核心角色，分别建一条旅程。
不要给所有角色都建旅程，会失焦。

**选择标准**：使用频率最高 + 业务影响最大的角色优先。

### Step 2：列阶段

按 6 阶段模板，每个阶段写：
- `id`：JS-001 起编号
- `name`：阶段名
- `description`：用户在这一阶段试图做什么
- `role_ids`：参与的角色

跨角色的阶段（如「审批」涉及两个角色），role_ids 写多个。

### Step 3：标关键路径

每个阶段标 1-3 个关键任务（KeyTask 引用 Step 2 输出）：
- 「关键」= 用户高频做 + 业务影响大 + 容易出问题

### Step 4：标情绪曲线（可选）

如果你有用户访谈数据，给每个阶段标情绪：`positive / neutral / negative / critical_pain`。
这有助于后续判定问题严重等级。

### Step 5：标体验机会点

每个阶段写 1-2 个「最容易出体验问题的位置」，帮助后续 task-generation 阶段聚焦。

## 模板

```yaml
journey_map:
  primary_role: R-001
  journey_name: 端到端规则上线
  stages:
    - id: JS-001
      name: 进入与定位
      description: 运营专员登录后找到规则配置入口
      role_ids: [R-001]
      key_task_ids: [KT-001-01]
      emotion: neutral
      pain_hotspots:
        - 规则配置入口埋藏太深，要点 3 次菜单
    - id: JS-002
      name: 前置配置
      description: 选择数据集 + 选模板
      role_ids: [R-001]
      key_task_ids: [KT-001-02]
      emotion: negative
      pain_hotspots:
        - 数据集列表加载慢
        - 模板预览不直观
    - id: JS-003
      name: 任务执行
      description: 编辑规则字段 + 测试规则
      role_ids: [R-001]
      key_task_ids: [KT-001-03]
      emotion: critical_pain
      pain_hotspots:
        - 规则测试结果与生效结果有差异
    - id: JS-004
      name: 结果理解
      description: 查看规则生效后的数据分布
      role_ids: [R-001, R-003]
      key_task_ids: [KT-001-04]
      emotion: neutral
    - id: JS-005
      name: 报告决策
      description: 基于分布决定是否调整或回滚
      role_ids: [R-001, R-002]
      emotion: negative
      pain_hotspots:
        - 调整与回滚都没有版本对比
    - id: JS-006
      name: 闭环优化
      description: 把高频规则沉淀为模板供复用
      role_ids: [R-001]
      emotion: positive
```

## 示例（从分类分级项目提炼）

```text
角色：运营专员（创建规则）+ 数据管理员（审批）
旅程：端到端规则上线（v1.0）

JS-001 进入与定位
  → 入口被埋在三级菜单（pain）
  → 需要记住跟「数据治理」混在一起
  
JS-002 前置配置
  → 数据集列表 5s+ 加载（pain）
  → 选错数据集后切换没有警告
  
JS-003 任务执行 ★ critical
  → 规则测试结果与最终生效不一致（pain）
  → 没有「保存草稿」（pain）
  → 多人协作时无锁定机制
  
JS-004 结果理解
  → 数据分布图加载慢
  → 不能下载结果
  
JS-005 报告决策
  → 版本对比要回到列表页
  → 回滚没有二次确认
  
JS-006 闭环优化
  → 模板沉淀流程不清晰
```

## 注意事项

### 不要做的事

- ❌ **按菜单顺序建阶段**：「我的工作台 → 数据管理 → 设置」是 IA，不是旅程
- ❌ **每个角色建独立完整旅程**：跨角色协作的关键，旅程要交织
- ❌ **超过 8 个阶段**：粒度太细
- ❌ **跳过情绪曲线就标问题**：没有情绪基线，严重等级判定容易飘

### 需要做的事

- ✅ 至少 1 条跨角色旅程（哪怕是「提交 → 审批」两步）
- ✅ 每个 pain_hotspot 后续都要在 task-generation 中产生对应任务
- ✅ 旅程的命名用「动词 + 名词」（「端到端规则上线」），不用「XX 模块使用」

## 与其他 stage 的衔接

- 上游：`prd-understanding` 提供 modules / roles / scenarios
- 下游：`task-generation` 基于阶段 + pain_hotspots 拆任务
- 下游：`issue-attribution` 用 journey_stage_id 做问题归因
