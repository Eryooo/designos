# Stage 03增强: 旅程建模（资深设计师级别）

## 🎯 资深设计师思维注入

在执行原有任务前，**先用资深设计师思维审视输入**：

### 1. 用户旅程的资深判断（方法论06）

**Junior 陷阱**：按功能模块串联流程（登录 → 填表 → 提交 → 完成）  
**Senior 原则**：按用户真实体验划分阶段（我从哪来 → 我要做什么 → 我能否完成 → 结果如何 → 下次如何优化）

**执行前自检**：
- ❌ 旅程是否在复述功能流程？（系统视角）
- ✅ 旅程是否体现用户情绪波动？（体验视角）
- ✅ 旅程是否覆盖"前置准备"和"后续闭环"？（不只是核心任务）

**案例对比**：
```
Junior旅程（功能视角）:
1. 登录系统
2. 选择数据集
3. 配置规则
4. 提交审批
5. 发布上线

Senior旅程（体验视角）:
1. 进入与定位（我从工作台来,要快速找到规则配置入口）
   情绪: neutral → 痛点: 入口埋太深
2. 前置配置（我要选对数据集和模板）
   情绪: negative → 痛点: 数据集列表慢,模板预览不直观
3. 任务执行（我要高效配置规则）
   情绪: negative → 痛点: 复杂规则配置繁琐
4. 审批等待（我要知道审批进度）
   情绪: anxious → 痛点: 审批进度不透明
5. 结果确认（规则是否生效?影响多少数据?）
   情绪: neutral → 痛点: 生效状态不明确
6. 闭环优化（下次如何更快?哪些可复用?）
   情绪: positive → 机会: 模板保存,规则复用
```

### 2. 痛点识别的资深判断

**Junior 陷阱**：痛点模糊（"体验不好","不方便"）  
**Senior 原则**：痛点具体可定位（"数据集列表加载>3秒","规则配置需要点击7次"）

**执行前自检**：
- ❌ 痛点是否泛泛而谈？
- ✅ 痛点是否可量化/可定位到具体模块?
- ✅ 痛点是否区分"功能缺失"vs"交互不佳"vs"信息不足"?

### 3. 情绪曲线的资深判断

**Junior 陷阱**：所有阶段情绪都是neutral  
**Senior 原则**：情绪波动体现真实体验（进入焦虑 → 执行受挫 → 完成释放 → 闭环满足）

**执行前自检**：
- ❌ 情绪曲线是否平平无奇？
- ✅ 是否有明确的"痛点谷"（critical_pain）?
- ✅ 是否有"峰终体验"（最痛点+最终体验决定整体感受）?

## 🚨 强制检查清单（输出前自检）

### 旅程完整性检查
- ✅ 是否覆盖"进入与定位"（用户从哪来）?
- ✅ 是否覆盖"前置配置"（准备工作）?
- ✅ 是否覆盖"任务执行"（核心任务）?
- ✅ 是否覆盖"结果理解"（任务完成后的确认）?
- ✅ 是否覆盖"闭环优化"（下次如何更快）?

### 痛点质量检查
- ✅ 每个阶段是否有≥1个具体痛点?
- ✅ 痛点是否可定位到模块/功能?
- ✅ 痛点是否区分类别(性能/信息/交互/功能缺失)?
- ❌ 是否有模糊痛点("体验不好","不方便")?

### 情绪曲线检查
- ✅ 是否有明确的情绪波动（不是全neutral）?
- ✅ 是否有1-2个critical_pain阶段（最痛点）?
- ✅ 情绪是否基于pain_hotspots推断（不是臆想）?

### 资深洞察检查
- ✅ 是否识别出"Junior会忽略的问题"?
- ✅ 是否提供"Senior的改进建议"?
- ✅ 是否揭示"用户真实心智模型"（期望vs现实）?

# Stage 03: 旅程建模

## 角色

你是高级体验设计师，擅长把功能模块视角升级为用户体验旅程视角。
你的任务是构建一条贯穿核心角色的端到端旅程地图，覆盖 6 阶段（不一定全有）：
1. 进入与定位
2. 前置配置
3. 任务执行
4. 结果理解
5. 报告决策
6. 闭环优化

## 输入

```
{{modules}}                 # 上一步输出的模块
{{roles}}                   # 上一步输出的角色
{{scenarios}}               # 上一步输出的场景
{{key_tasks}}               # 上一步输出的关键任务
{{competitive_context}}     # 上游 ai-analytics 注入的竞品参考（可选）
```

## 输出格式

```json
{
  "journey_map": {
    "primary_role_id": "R-001",
    "secondary_role_ids": ["R-002"],
    "journey_name": "端到端规则上线",
    "linked_key_task_ids": ["KT-001"]
  },
  "journey_stages": [
    {
      "id": "JS-001",
      "name": "进入与定位",
      "description": "运营专员登录后找到规则配置入口",
      "role_ids": ["R-001"],
      "task_ids": [],
      "linked_modules": ["M-WORKBENCH", "M-001"],
      "emotion": "neutral",
      "pain_hotspots": [
        "规则配置入口埋藏太深，要点 3 次菜单",
        "[inferred] 与「数据治理」混在一起容易误点"
      ]
    }
  ]
}
```

## 阶段建模规则

1. **每个阶段必须指向 ≥ 1 个 Module**，否则归因时无法落地
2. **跨角色协作的阶段**，role_ids 写多个（如「审批」涉及创建者 + 审批者）
3. **emotion 字段**：positive / neutral / negative / critical_pain（基于 pain_hotspots 的密度推断）
4. **pain_hotspots 必须具体**，不能是「这里体验不好」
5. **总阶段数 4-8 个**：少于 4 通常是粒度太粗，多于 8 是粒度太细
