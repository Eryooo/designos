## 1. 项目背景

数据治理团队需要一套规则配置平台，让运营专员能够独立创建、维护数据分类规则，
并通过审批流程确保规则质量，最终应用到全量业务数据。

当前业务痛点：
- 规则配置依赖工程师写 SQL，运营业务方迭代慢
- 规则版本无管理，回滚困难
- 规则审批流程混乱，常常误上线

## 2. 业务目标

为数据治理团队提供数据分类规则配置能力，让规则上线效率提升 3 倍，
版本切换错误率下降至 < 1%。

## 3. 目标用户

| 角色 | 主要职责 |
|---|---|
| 运营专员 | 创建、调整、维护数据分类规则 |
| 数据管理员 | 审批规则、监控规则生效情况 |

## 4. 功能模块

### 4.1 规则草稿（M-001）

**核心功能**：
- 创建规则（支持 15+ 字段）
- 草稿保存（v1.0 暂不支持自动保存）
- 规则模板复用
- 多人协作（v1.0 暂不支持锁定机制）

**页面**：`/rules/new`、`/rules/edit/:id`、`/rules/draft`

### 4.2 版本管理（M-002）

**核心功能**：
- 查看规则历史版本
- 版本切换
- 回滚到历史版本
- 版本对比（v1.1 计划，v1.0 暂不支持）

**页面**：`/rules/versions`、`/rules/version/:id`

### 4.3 审批流（M-003）

**核心功能**：
- 提交规则审批
- 审批人查看规则
- 审批通过 / 驳回
- 审批通知（邮件 + 站内）

**页面**：`/approval/list`、`/approval/detail/:id`

## 5. 用户场景

### 场景 1：新增分类规则

**触发**：业务方提出新分类口径（如「将订单按地域和金额双维度分类」）
**路径**：M-001（创建草稿）→ M-003（提交审批）→ 系统应用
**期望**：规则在 1 个工作日内上线

### 场景 2：调整现有规则

**触发**：业务规则变化或发现规则误差
**路径**：M-002（找到当前规则）→ M-001（编辑）→ M-003（审批）→ M-002（版本切换）

### 场景 3：紧急回滚

**触发**：发现规则导致数据错误
**路径**：M-002（查看历史）→ 一键回滚

## 6. 业务流程

```
[运营专员] → 创建规则 → 提交审批
                              ↓
[数据管理员] → 查看规则 → 审批通过 / 驳回
                              ↓ (通过)
                       系统应用规则到全量数据
```

## 7. v1.0 范围

**包含**：
- 规则草稿、版本管理、审批流
- Web 后台
- 邮件通知

**不包含**（v1.1+）：
- 自动保存草稿
- 多人协作锁定
- 规则市场（共享他人规则）
- AI 规则推荐
- 移动端
