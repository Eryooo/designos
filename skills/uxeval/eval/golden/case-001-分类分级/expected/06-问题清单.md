# 期望产物：问题清单（节选）

> 这是 case-001 期望的代表性问题（不是全部，golden 测试只验证关键问题是否被识别）。
> 完整问题数量 15-35 之间均可。

## I-001 工作台规则待办与其他待办无视觉区分

| 字段 | 值 |
|---|---|
| 严重等级 | major |
| 启发式原则 | S2 再认而非记忆, P3 一致性与标准化 |
| 旅程阶段 | JS-001 进入与定位 |
| 任务 | T-001 |
| 模块 | M-WORKBENCH |
| 基准来源 | screenshot |

**描述**：在「早晨查看待办」场景下，运营专员进入工作台，需要从 12 条混合待办中找出规则相关；当前所有待办字号、颜色、图标相同，平均需 8 秒才能定位 5 条规则相关待办。

**用户影响**：每天早晨运营专员需 8-15 秒定位规则待办，且漏看率约 23%。高频场景下累计认知负担显著。

**改进建议**：建议为规则相关待办增加视觉标识（左侧蓝色侧边条 + 「规则」标签），与其他待办形成视觉层级。当前违反 S2（需逐字阅读判断类型）和 P3（所有待办视觉权重相同）。

## I-005 规则编辑草稿丢失（无自动保存）

| 字段 | 值 |
|---|---|
| 严重等级 | critical |
| 启发式原则 | S1 用户控制度与自由度, F1 预防出错 |
| 旅程阶段 | JS-003 任务执行 |
| 任务 | T-005 |
| 模块 | M-001 |
| 基准来源 | screenshot |

**描述**：在编辑长规则（15+ 字段）时，如关闭浏览器或刷新页面，所有未提交的内容全部丢失。无自动保存、无离开警告、无草稿恢复。

**用户影响**：在「编辑长规则中途被打断」场景下（每条规则至少发生 1 次，常发生于会议、电话），运营专员所有 15 个字段的输入全部丢失，平均损失工作时长 25 分钟，约 60% 用户经历过 ≥ 1 次完整丢失。

**改进建议**：建议为规则编辑表单增加「每 30 秒自动保存草稿」+ 「关闭页面提示「确定离开？未保存的内容会丢失」」。当前违反 S1（用户无法保留中途状态）和 F1（应在错误发生前阻止，而非接受丢失）。

## I-007 规则保存提交无加载反馈

| 字段 | 值 |
|---|---|
| 严重等级 | major |
| 启发式原则 | F2 系统状态可见性 |
| 旅程阶段 | JS-003 任务执行 |
| 任务 | T-007 |
| 模块 | M-001 |
| 基准来源 | screenshot |

**描述**：在「保存规则草稿」场景下，运营专员点击「保存」后，按钮无 loading 状态、无消息提示、无页面变化；约 3-5 秒后才跳转。

**用户影响**：高频场景（每条规则平均保存 5+ 次），运营专员等待 3-5 秒无反馈，约 40% 用户在前 3 次使用中重复点击保存按钮。

**改进建议**：建议把「保存」按钮从无反馈改为「点击后立即变 loading 状态（按钮文案「保存中...」+ 禁用）+ 完成后顶部 toast 提示「保存成功」」。当前违反 F2。

## I-012 规则测试结果与生效结果不一致

| 字段 | 值 |
|---|---|
| 严重等级 | critical |
| 启发式原则 | F2 系统状态可见性, F5 对象关系可理解性 |
| 旅程阶段 | JS-003 任务执行 |
| 任务 | T-012 |
| 模块 | M-001 |
| 基准来源 | screenshot |

**描述**：规则测试时显示生效条数 87，实际上线后是 92 条。测试样本与生效逻辑存在差异，但页面无任何说明。

**用户影响**：运营专员根据测试结果做决策，但实际数据偏差 5%+，可能导致错误规则上线。在监管类场景中存在合规风险。

**改进建议**：建议在测试结果旁明确标注「测试基于近 30 天采样数据，与全量生效可能有 5% 偏差」+ 提供「使用全量数据测试」选项。当前违反 F2（用户对系统状态有错误预期）和 F5（数据来源 / 偏差不透明）。

## I-014 版本回滚无影响范围预览

| 字段 | 值 |
|---|---|
| 严重等级 | critical |
| 启发式原则 | S1 用户控制度与自由度, F1 预防出错 |
| 旅程阶段 | JS-005 报告决策 |
| 任务 | T-014 |
| 模块 | M-002 |
| 基准来源 | screenshot |

**描述**：版本回滚弹窗只有「确认 / 取消」，没有显示回滚后会影响的数据条数、对比当前版本的差异、潜在影响。

**用户影响**：在「紧急回滚」场景下（高风险操作），用户在不知道影响范围的情况下点确认，可能造成更大数据问题。

**改进建议**：建议把回滚确认弹窗从「极简确认」改为「展示影响范围（条数、关键差异、估算时长）+ 二次确认（输入版本号）」。违反 F1（高危操作应有错误预防）和 S1（用户应有完整信息再做决策）。
